RAG 的核心目标是让大模型先检索外部知识，再基于检索结果生成回答，从而减少幻觉并提升事实性。

知识库工程通常不只是“把文档塞进向量库”，而是要处理文档清洗、切分、索引、召回、重排、更新和评估等全流程。

Document Loading 是把 PDF、Word、网页、Markdown、HTML 等数据源读取成统一文本或结构化对象的过程。

文本清洗的目标是去掉噪声字符、重复内容、无意义空白、乱码和格式污染，提升后续切分和检索质量。

Chunking 是将长文档切分成适合检索和向量化的小片段，是知识库构建的基础步骤。

Chunk 太大容易带来噪声和召回不准，Chunk 太小又容易丢失上下文，因此切分需要兼顾语义完整性和检索粒度。

Chunk overlap 是相邻文本块之间保留重叠内容的策略，用来减少切分造成的信息断裂。

按标题、段落、列表层级切分，通常比纯按固定长度切分更符合语义边界。

Parent-Child Chunking 是一种分层切分方式，小块用于召回，大块用于最终上下文补充，适合长文档场景。

Embedding 是把文本映射为向量表示的过程，语义相近的文本在向量空间中距离更近。

Embedding 模型的效果会直接影响召回质量，因此在知识库系统中属于关键基础设施。

向量检索依赖向量相似度计算，常见做法是先把查询和文档都转成 embedding，再做近邻搜索。

Cosine Similarity 常用于文本语义相似度计算，因为它更关注向量方向而不是长度。

Dot Product 在向量归一化后也常被用于相似度排序，很多向量检索引擎会采用这种方式。

Euclidean Distance 更强调几何距离，在文本检索中相对少见，但在某些向量空间任务里也会用到。

BM25 是经典关键词检索算法，擅长处理显式关键词和术语匹配，对技术文档尤其有用。

Hybrid Retrieval 是将关键词检索和向量检索结合起来的方法，常用于提升召回率和鲁棒性。

Hybrid Retrieval 的优势在于既能抓住语义相似，也能抓住术语精确匹配，适合企业知识库和技术问答。

Recall 阶段追求“尽量找全相关内容”，Rank 阶段追求“把最相关内容排前面”，两者目标不同。

Rerank 是对初步召回结果进行二次排序的过程，通常用更强的模型来提升精度。

Query Rewrite 是把用户原始问题改写成更适合检索的形式，常用于提升模糊问题的召回效果。

Query Understanding 包括识别用户意图、关键词、实体、约束条件和上下文依赖，是检索前的重要步骤。

Metadata 是附加在文档片段上的结构化信息，比如来源、标题、页码、时间、标签和权限信息。

Metadata 不仅有助于过滤和排序，也有助于来源追踪、权限控制和结果解释。

Index 构建通常不只是向量索引，还可能包括倒排索引、元数据索引和分区索引。

向量库不是知识库本身，而是知识库的存储和检索基础设施，业务逻辑通常还在上层服务处理。

Milvus、Chroma、FAISS 等都是常见的向量检索技术或工具，选型要结合规模、部署方式和运维成本。

文档更新后如果不及时同步索引，容易召回旧内容，因此增量更新和索引一致性很重要。

增量入库通常只处理新增、修改或删除的文档，而不是每次全量重建。

知识库里需要处理重复文档、近重复文档和低质量文档，否则会明显影响检索效果。

去重既可以基于文本相似度，也可以基于结构化指纹、哈希和来源规则。

权限控制在企业知识库里很重要，不同用户应只能检索自己有权限访问的文档。

基于权限的检索过滤通常应该在召回前或召回后尽早做，避免敏感内容进入上下文。

Long Context 不等于高质量，上下文越长，噪声越多，模型越可能偏离重点。

Context Packing 的核心是把最相关、最有价值的片段组织成适合模型输入的上下文。

上下文构造时通常要考虑片段顺序、来源可信度、相关性和信息互补性。

RAG 系统中，答案是否准确不仅取决于模型能力，也取决于检索结果是否足够相关和完整。

Grounding 是让模型回答尽量基于检索到的证据，而不是依赖模型参数中的模糊记忆。

Source Attribution 可以提升回答可信度，尤其适合企业知识库、合规场景和高风险问答。

Hallucination 是模型在证据不足时生成看似合理但不真实的内容，是 RAG 要重点缓解的问题。

减少幻觉的常见手段包括检索增强、重排、引用来源、限定回答边界和不确定时保守输出。

知识库中的文档格式多样，PDF、网页、表格、图片和 Office 文件通常要用不同的解析策略。

PDF 解析时要特别注意版面信息、表格、页眉页脚和双栏排版，否则容易丢失结构。

OCR 通常用于图片或扫描件文本抽取，但 OCR 结果会引入噪声，因此后续清洗很重要。

表格类数据如果直接扁平化成纯文本，容易丢失行列关系，因此最好保留表格结构信息。

针对 FAQ、制度文档、产品手册和技术规范，不同类型文档的切分和检索策略通常不一样。

知识库评估不能只看“有没有答案”，还要看召回是否准确、上下文是否相关和最终回答是否可信。

常见的离线评估指标包括 Recall@K、MRR、NDCG、Answer Relevancy 和 Faithfulness。

Recall@K 关注前 K 条召回中是否包含正确答案，是检索系统最基础的指标之一。

MRR 关注第一个正确结果出现的位置，越靠前说明排序越好。

Answer Relevancy 主要衡量回答是否与问题相关，适合评估生成阶段质量。

Faithfulness 更关注答案是否忠实于检索证据，适合衡量幻觉风险。

知识库评估通常需要标准问答集，而不是只靠人工感觉判断。

线上知识库系统还要关注延迟、吞吐、缓存命中率和更新时效性，这些都是工程指标。

检索质量不仅取决于 embedding 和向量库，还取决于文档切分、查询改写、召回策略和重排策略的整体设计。

知识库工程里，最重要的不是“存了多少文档”，而是“能不能稳定找到对的文档”。

分层检索通常指先做粗召回，再做精排，必要时回到更大上下文补充信息。

多阶段召回可以先用关键词和语义召回扩大候选集，再用 rerank 筛掉噪声。

如果用户问题很短、很模糊，直接检索往往效果不好，通常需要先做问题补全或意图识别。

多轮对话场景下，当前问题常常依赖历史上下文，所以检索查询不能只看这一轮文本。

对话式 RAG 一般会把最近轮次、会话摘要或关键信息拼入检索查询中。

会话摘要可以帮助控制上下文长度，但摘要质量不足时也可能误导检索。

历史上下文不应该无限堆积，否则会引入噪声和 token 成本问题。

在知识库问答里，检索结果排序通常比召回数量更重要，因为最终能喂给模型的上下文有限。

如果召回结果过多，模型容易被无关内容干扰，导致回答偏题或混淆多个来源。

如果召回结果过少，模型可能缺少足够证据，容易出现幻觉或回答不完整。

Chunk 粒度设计通常要结合文档类型调整，制度类、FAQ 类、教程类和论文类文档的切分方式会不同。

FAQ 类文档通常适合较小粒度的独立问答块，而长教程更适合按章节或语义段落切分。

对于包含列表、步骤、定义和例子的文档，切分时应尽量保留完整结构。

如果 chunk 的语义边界被破坏，embedding 往往无法准确表达该片段的真实主题。

文档结构化处理比简单抽纯文本更利于后续检索和引用。

表格内容最好既保留原始结构，又能生成适合语义检索的文本描述。

图片中的文本通常需要 OCR，但 OCR 结果要配合版面分析和纠错，否则召回质量会受影响。

在企业知识库中，页面级别、段落级别和表格级别的索引通常可以并存。

元数据可以用于按部门、权限、时间、版本和主题过滤结果。

知识库系统里，版本管理很关键，因为同一份制度、同一款产品或同一篇文档可能存在多个版本。

如果不做版本控制，用户可能检索到过期内容，尤其在制度类和产品类知识库里风险较高。

增量更新时通常要处理新增、修改和删除三种情况，不能只做新增。

删除文档后，如果索引没有同步清理，仍可能召回已失效内容。

近重复文档会造成召回偏置，导致结果集中在少数相似片段上。

近重复去重可以利用文本相似度、hash 指纹或结构化规则实现。

知识库内容清洗不仅包括文本清洗，还包括去广告、去页眉页脚、去导航和去噪声片段。

如果文档来源多样，清洗策略往往需要按来源类型分别定制。

企业级知识库通常要考虑权限隔离，不能让普通用户检索到无权限文档。

权限控制一般要和检索流程结合，而不是只在前端隐藏文档名称。

在检索阶段做权限过滤，往往比生成后再过滤更安全。

如果检索结果来自多个来源，最好保留来源信息，方便后续核验和引用。

上下文拼接时，来源顺序、相关性和互补性都会影响最终答案质量。

片段排序通常会优先考虑与问题强相关的内容，再补充必要背景信息。

如果把多个弱相关片段一起塞给模型，可能会让回答逻辑变乱。

RAG 不是简单的“先搜再答”，而是“检索、筛选、组织、生成”的组合流程。

生成阶段最好明确要求模型只基于检索证据回答，避免自由发挥过多。

如果答案必须可追溯，模型输出中应带上引用片段或来源标识。

Grounded Generation 的核心是让模型尽可能贴着证据说话，而不是进行无约束创作。

在知识库问答中，有时宁可少答一点，也不要回答得很满但不可靠。

如果检索不到足够依据，系统应当明确提示信息不足，而不是强行输出。

Query Rewrite 不只是改写措辞，还可能补全实体、消歧和标准化用户表达。

对于口语化查询、简称查询和上下文缺失查询，改写通常能显著改善召回。

如果用户输入中包含多个意图，最好先拆成子查询再分别召回。

多意图查询直接检索，往往会把不同主题的信息混在一起。

召回策略可以分成稀疏召回、稠密召回和混合召回三类。

稀疏召回更擅长关键词匹配，稠密召回更擅长语义匹配，混合召回兼顾两者。

在技术文档和企业文档场景里，混合召回通常是更稳妥的默认选择。

Rerank 模型的输入一般是“问题 + 候选片段”，它负责判断候选是否真正相关。

如果 rerank 模型太慢，整个 RAG 系统延迟会明显上升，因此需要平衡效果和速度。

有些场景会对候选先做轻量过滤，再对较少的候选做重排，以控制成本。

知识库工程还需要关注缓存设计，例如检索缓存、重排缓存和回答缓存。

缓存命中率高时能明显降低延迟，但缓存策略必须考虑文档更新和用户权限变化。

如果知识更新频繁，缓存 TTL 不能过长，否则容易返回旧答案。

上线后的知识库系统最好有监控指标，比如召回命中率、无答案率、平均检索时延和用户追问率。

追问率高并不一定是坏事，但如果同一类问题反复追问，可能意味着检索质量不足。

无答案率升高通常说明知识覆盖不足、查询改写不够好，或者切分策略有问题。

离线评估和线上反馈要结合起来看，单一指标很难完整反映知识库质量。

标准问答集应该覆盖常见问法、变体问法和模糊问法，不能只用“标准答案”测试。

如果只测单轮准确率，很难发现多轮上下文、权限、版本和噪声带来的真实问题。

面向生产的知识库评估，最好同时看检索、重排和生成三个环节。

检索层看召回率和排序质量，生成层看忠实度、完整性和可读性。

知识库工程里，最常见的问题往往不是模型不够强，而是数据、切分和检索链路设计不合理。

问题：什么是 RAG？ 答案：RAG 是 Retrieval-Augmented Generation 的缩写，指在大模型生成答案前先从外部知识库中检索相关内容，再结合检索结果进行回答。它的核心价值是减少幻觉、提升事实性和增强知识更新能力。

问题：为什么 RAG 在知识库问答里很常见？ 答案：因为大模型参数中的知识是有限的，而且更新不够实时。RAG 可以把最新、最相关的外部知识引入回答过程，因此特别适合企业知识库、文档问答和内部问答场景。

问题：RAG 的基本流程是什么？ 答案：通常包括文档加载、清洗、切分、向量化、索引构建、查询检索、重排、上下文拼接和最终生成。这个流程的目标是让模型基于检索到的证据来回答。

问题：为什么知识库工程不能只做“向量入库”？ 答案：因为真正影响效果的环节很多，包括文档解析、切分方式、元数据设计、检索策略、重排策略、权限控制和评估体系。只做入库远远不够。

问题：什么是 Chunking？ 答案：Chunking 是把长文档切成多个适合检索和向量化的片段。合理的切分能提高召回精度和上下文利用效率。

问题：Chunk 太大或太小分别会有什么问题？ 答案：Chunk 太大容易带来噪声和语义混杂，Chunk 太小又会让上下文不完整，影响理解和检索。通常要在语义完整性和检索粒度之间平衡。

问题：为什么很多知识库会使用 chunk overlap？ 答案：因为相邻片段之间保留重叠内容，可以减少切分导致的上下文断裂。这样有助于模型理解前后关联，尤其适合长文档。

问题：什么是 Parent-Child Chunking？ 答案：Parent-Child Chunking 是一种分层切分策略，小块用于检索，大块用于最终回答补充上下文。它特别适合长文档和结构复杂的资料。

问题：什么是 Embedding？ 答案：Embedding 是将文本转换成向量表示的过程。语义相近的文本在向量空间中通常距离更近，因此适合用于语义检索。

问题：为什么 Embedding 对 RAG 很重要？ 答案：因为向量检索的质量直接依赖 embedding 的表达能力。如果 embedding 表达不准，系统就很难召回真正相关的内容。

问题：什么是向量检索？ 答案：向量检索是把查询和文档都转成向量，再根据相似度寻找最相关的文档片段。它能找到词面不同但语义相近的内容。

问题：Cosine Similarity 在检索里有什么作用？ 答案：Cosine Similarity 常用来衡量两个向量的方向相似度，数值越高表示越相近。在文本语义检索中它非常常见。

问题：BM25 是什么？ 答案：BM25 是一种经典的关键词检索排序算法，主要根据词项匹配和词频进行打分。它在术语明确、关键词敏感的场景中效果很好。

问题：为什么很多 RAG 系统会做混合检索？ 答案：因为向量检索更擅长语义匹配，BM25 更擅长关键词精确匹配。混合检索可以同时兼顾两种能力，召回更稳。

问题：什么是 Query Rewrite？ 答案：Query Rewrite 是将用户原始问题改写为更适合检索的形式。它常用于口语化提问、上下文不完整或表达模糊的场景。

问题：什么是 Query Understanding？ 答案：Query Understanding 是理解用户问题意图、关键词、实体和约束条件的过程。它是决定后续检索和生成策略的重要前置步骤。

问题：为什么多轮对话场景下要考虑历史上下文？ 答案：因为当前问题常常依赖前文信息，如果不把历史纳入检索，系统容易误解用户意图。对话式 RAG 一般会结合会话摘要或最近轮次一起检索。

问题：什么是 Rerank？ 答案：Rerank 是对初步召回结果进行二次排序的过程，通常使用更强的模型判断候选片段是否真正相关。它可以明显提升最终上下文质量。

问题：为什么检索后常常还要重排？ 答案：因为初步召回阶段强调覆盖率，候选中可能含有很多噪声。重排可以帮助筛掉不相关内容，提高最终回答的准确性。

问题：什么是 Metadata？ 答案：Metadata 是附加在文档片段上的结构化信息，比如来源、标题、页码、时间、标签和权限。它能帮助检索过滤、结果排序和来源追踪。

问题：Metadata 在知识库系统里有什么价值？ 答案：它不仅能帮助按时间、权限、主题过滤文档，还能提高可解释性和可维护性。对于企业知识库来说非常重要。

问题：为什么 PDF 解析在知识库里很重要？ 答案：因为很多企业文档以 PDF 形式存在，而且 PDF 可能包含复杂排版、表格和页眉页脚。解析质量不好会直接影响后续切分和召回。

问题：OCR 在 RAG 里有什么作用？ 答案：OCR 用于识别图片或扫描件中的文字，是处理非结构化文档的重要方式。很多图片化 PDF 和扫描文档都依赖 OCR 才能进入知识库。

问题：为什么表格文档不能简单按纯文本处理？ 答案：因为表格的行列关系会被破坏，导致语义丢失。更好的做法是保留结构信息，同时生成适合检索的文本描述。

问题：知识库中为什么要做去重？ 答案：因为重复或近重复文档会让检索结果偏向少数内容，增加噪声并浪费索引空间。去重有助于提升检索多样性和准确性。

问题：文档更新后为什么要同步索引？ 答案：如果索引不同步，系统可能仍然召回旧版本内容，导致回答不准确。对于制度类、产品类和知识变化快的场景，这个问题尤其重要。

问题：什么是增量更新？ 答案：增量更新是只处理新增、修改或删除的文档，而不是每次全量重建索引。它更适合持续更新的知识库系统。

问题：什么是权限过滤？ 答案：权限过滤是根据用户身份和权限，只让系统召回有权限访问的文档。企业知识库里这是必须考虑的安全问题。

问题：为什么权限控制最好在检索阶段就做？ 答案：因为如果无权限内容先进入上下文，再在后面过滤，存在泄露风险。越早过滤越安全。

问题：什么是 Grounding？ 答案：Grounding 指让模型回答尽量建立在检索到的证据上，而不是凭空生成。它是降低幻觉的重要思路。

问题：什么是 Source Attribution？ 答案：Source Attribution 是在回答中标明信息来源，帮助用户判断答案来自哪里。它能提升可信度和可追溯性。

问题：什么是 Hallucination？ 答案：Hallucination 指模型生成了看起来合理但事实上不准确的内容。RAG 的一个重要目的就是缓解这个问题。

问题：如何减少 RAG 中的幻觉？ 答案：常见方法包括提高检索质量、加入重排、限制回答边界、保留来源信息，以及在不确定时保守回答。工程上通常要多种方法结合。

问题：什么是上下文拼接？ 答案：上下文拼接是把检索到的片段按一定规则组织起来输入给模型。拼接顺序、长度和相关性都会影响最终回答效果。

问题：为什么不能把所有召回结果都塞给模型？ 答案：因为上下文窗口有限，而且过多内容会引入噪声，反而降低回答质量。通常要筛选最相关的少量片段。

问题：什么是 Context Packing？ 答案：Context Packing 是把最相关的内容压缩、排序并组织成适合模型输入的上下文。它的目标是让有限上下文发挥最大作用。

问题：什么是 Recall@K？ 答案：Recall@K 是衡量前 K 个召回结果中是否包含正确答案的指标。它常用于评估检索阶段的覆盖能力。

问题：什么是 MRR？ 答案：MRR 是 Mean Reciprocal Rank，用来衡量第一个正确结果出现得有多靠前。排名越靠前，MRR 越高。

问题：什么是 NDCG？ 答案：NDCG 是一种排序评估指标，用于衡量结果排序是否更符合相关性。它常用于检索和排序任务评估。

问题：什么是 Faithfulness？ 答案：Faithfulness 主要衡量模型回答是否忠实于检索证据。它特别适合评估 RAG 中的生成是否出现“编造”。

问题：什么是 Answer Relevancy？ 答案：Answer Relevancy 衡量答案和问题是否相关，是否真正回答了用户想问的内容。它常用于生成质量评估。

问题：为什么知识库评估不能只看人工感觉？ 答案：因为人工感觉不稳定，而且难以量化和回归。标准评估集和指标能帮助系统持续优化和对比不同方案。

问题：什么是离线评估？ 答案：离线评估是先用标准数据集和固定指标测试系统效果，不依赖线上真实用户流量。它适合做方案比较和回归测试。

问题：线上评估通常看哪些指标？ 答案：线上一般看检索延迟、生成延迟、无答案率、追问率、用户反馈和命中率等指标。它们能反映真实使用体验。

问题：为什么知识库系统要关注延迟？ 答案：因为检索、重排和生成都会影响用户体验。知识库系统如果响应太慢，即使答案质量高，也会影响实际可用性。

问题：什么是缓存命中？ 答案：缓存命中是指系统直接从缓存中拿到结果，而不是重新检索或重新计算。它能有效降低延迟和资源消耗。

问题：知识库系统里可以缓存哪些内容？ 答案：常见可缓存内容包括检索结果、重排结果、答案结果和部分中间步骤结果。缓存可以显著提升高频查询的性能。

问题：为什么知识库系统不能只靠缓存？ 答案：因为知识会更新，缓存过久会导致旧答案被重复返回。缓存必须和知识版本、更新时间结合使用。

问题：什么是多阶段召回？ 答案：多阶段召回是先粗召回，再精筛，再重排的流程。它能在保证召回覆盖的同时控制噪声和延迟。

问题：为什么知识库系统要做监控？ 答案：因为只有监控关键指标，才能发现召回下降、延迟升高或无答案率增加等问题。没有监控就很难持续优化。

问题：知识库工程师最核心的能力是什么？ 答案：核心能力是把非结构化知识整理成可检索、可评估、可维护的系统，并持续提升召回质量、生成质量和工程稳定性。
