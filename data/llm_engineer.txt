Agent 的任务流程通常包括理解用户意图、判断是否需要工具、选择合适工具、执行工具调用、整合结果并输出最终答案。

工具调用和直接生成答案的区别在于，直接生成依赖模型参数记忆，而工具调用会把问题交给外部系统处理，准确性和实时性通常更高。

如果问题涉及实时信息、私有数据或精确计算，优先使用工具调用而不是让模型直接编造答案。

Agent 选择工具时，通常会先根据用户意图分类，再结合工具描述、输入参数和当前上下文决定调用哪个工具。

一个好的工具应该职责单一、输入清晰、输出稳定，这样 Agent 才容易学会正确使用它。

工具的输入输出最好结构化，尤其是输入参数要明确类型、字段名和含义，避免模型乱传参数。

在设计 Agent 工具时，最常见的做法是把复杂能力拆成多个小工具，而不是做一个什么都能干的大工具。

ReAct 的核心在于推理和行动交替进行，而不是先一次性想完所有步骤。

ReAct 流程中，模型先观察当前问题，再决定行动，拿到工具返回结果后继续下一步推理。

Prompt 的作用不只是让模型回答问题，还包括约束角色、规定格式、控制风格和指导思考路径。

系统提示词通常用来定义 Agent 的身份、目标、边界和安全规则，是整个应用行为约束的基础。

用户提示词应该尽量具体，因为任务描述越清楚，模型越容易稳定输出。

如果希望模型输出可被程序处理的结果，最好在 prompt 中明确要求结构化格式，比如固定字段或 JSON。

Few-shot 示例能显著提升模型在特定任务中的稳定性，尤其适合分类、抽取和格式化输出任务。

RAG 的基本流程是先检索，再生成，核心目标是让答案尽量建立在真实资料基础上。

RAG 系统里，检索质量往往比生成模型本身更重要，因为“找对资料”通常比“写得漂亮”更关键。

Chunk 切分策略会直接影响检索效果，切得太大容易噪声多，切得太小又容易丢失上下文。

Chunk overlap 的作用是保留相邻片段的上下文连续性，减少信息被切断的问题。

向量检索前要先做 embedding，因为只有把文本转成向量，系统才能进行语义相似度计算。

Embedding 模型的好坏会直接影响召回效果，不同领域的文本适合不同的 embedding 模型。

BM25 更适合关键词明确、术语敏感的任务，向量检索更适合语义相近但词面不同的任务。

混合检索可以同时兼顾关键词精确匹配和语义召回，适合技术面试这种既看术语又看语义的场景。

Rerank 通常放在初步召回之后，用来对候选结果进行二次排序，提升最终返回质量。

Query rewrite 的目的是把用户模糊的问题改写得更适合检索，尤其适合长问题、口语化问题和上下文不完整的问题。

Context packing 的重点不是把所有检索结果都塞进去，而是筛选最有用的信息并按逻辑组织好。

Document loader 的任务是把不同格式的文件统一转成可处理的文本内容。

如果文档中有标题、段落、页码等结构，尽量在加载阶段保留下来，后面做来源引用时会更方便。

Metadata 在 RAG 中很重要，因为它不仅能帮助过滤检索结果，还能帮助定位来源和增强可解释性。

Source attribution 能让回答更可信，尤其是在知识库问答或面试助手里非常有价值。

Hallucination 往往会在模型缺少上下文、检索失败或 prompt 约束不足时更容易出现。

减少幻觉的常见方法包括加强检索、限定回答范围、要求引用来源和对不确定内容保持保守。

Context window 不够时，系统需要做截断、摘要、分段检索或上下文压缩，否则模型容易丢信息。

Token 数量会影响调用成本和响应速度，因此在设计 prompt 和上下文时要控制长度。

Latency 过高时，用户体验会明显下降，所以 Agent 系统要尽量减少无效调用和冗余推理。

如果多个工具都能完成同一任务，最好让 Agent 优先选择最简单、最稳定、最便宜的那个。

Error handling 不能省略，因为模型调用、检索、文件读取和接口请求都可能失败。

Fallback strategy 的一个常见做法是：检索失败时给出基于当前上下文的保守回答，并明确提示信息不足。

Confidence score 可以帮助系统决定是否继续检索、是否需要追问用户，或者是否应该拒答。

Prompt injection 防护的重点是不要盲目相信用户输入，要把系统规则和用户内容分层处理。

Guardrails 不只是安全问题，也包括格式约束、内容边界和业务规则约束。

Task decomposition 常常先按“先分析、再检索、再生成、再校验”的顺序拆解。

复杂任务中，先规划再执行通常比边做边想更稳定，尤其适合多步骤 Agent。

如果一个问题需要多个知识点共同回答，先做子问题拆解通常能提升最终答案质量。

Memory 一般分为短期和长期两类，短期记住当前对话，长期记住跨会话的用户偏好。

短期记忆适合面试助手中的上下文连续追问，长期记忆更适合个性化建议和历史复盘。

在面试助手中，用户简历和 JD 更适合作为临时上下文，而不是长期记忆写进知识库。

Workflow 本质上是把复杂过程变成固定步骤或条件分支，适合做稳定的业务流程控制。

LangChain 更偏向组件拼装，适合快速搭建应用；LangGraph 更适合做有状态、有分支的复杂流程。

如果任务流程比较简单，LangChain 的 Agent 已经够用，不一定一开始就上 LangGraph。

Structured Output 可以显著降低后处理成本，因为程序可以直接读取字段而不是解析自然语言。

在抽取类任务中，要求模型输出固定 JSON 格式通常比纯文本回答更稳定。

如果模型经常输出格式错误，可以通过更严格的 prompt、示例和后处理校验来改善。

多轮对话中，上一轮回答会影响下一轮提问，所以上下文管理非常重要。

如果上下文太长，不一定要全塞进模型，可以只保留关键摘要和最近几轮最相关内容。

在知识库问答中，答案生成前最好先确认检索内容是否足够相关，否则容易胡编。

如果检索到的结果分散，可以先让模型做一次摘要整合，再进入最终回答阶段。

Agent 应用里，真正难的往往不是“让模型回答”，而是“让它按正确流程去做事”。

模型能力再强，如果工具设计混乱、prompt 不清晰、上下文组织差，最终效果也会很不稳定。

模型在工程落地中通常不是单纯追求“更聪明”，而是追求更稳定、更低延迟、更低成本和更易维护。

大模型工程师常见工作之一是把模型能力包装成稳定的服务接口，而不是直接把模型暴露给业务端。

推理服务和训练服务是不同的工程目标，前者更关注延迟、吞吐和资源利用率，后者更关注梯度更新和训练效率。

如果模型主要用于在线问答，通常更看重首 token 延迟、流式输出和并发吞吐。

如果模型主要用于离线任务，比如批量生成或文档处理，则更看重批处理能力和总体吞吐。

模型预热的目的是降低冷启动延迟，常见做法包括提前加载权重、初始化缓存和执行一次空请求。

GPU 上下文切换和显存碎片会影响推理性能，因此线上部署时要关注显存利用率和请求调度方式。

连续批处理可以把不同请求拼成一个批次一起推理，从而提高 GPU 吞吐率。

vLLM 的优势之一是针对在线推理做了更高效的 KV Cache 管理和调度优化。

TGI 更偏向开箱即用的推理部署方案，适合快速搭建生产级服务。

TensorRT-LLM 更强调极致推理性能优化，通常适合对延迟和吞吐要求更高的场景。

模型量化的核心目标是减少模型参数精度，从而降低显存占用和提升推理速度。

INT8 和 INT4 量化通常能明显减少资源消耗，但也可能带来一定精度损失。

FP8 是近年来常见的低精度推理格式之一，适合在支持该格式的硬件上提升性能。

量化后常见问题包括精度下降、输出不稳定、长文本性能变差和某些任务效果退化。

模型并不是越大越好，实际落地中常常需要在能力、成本和时延之间做平衡。

不同任务可以使用不同尺寸的模型，比如简单分类和抽取任务可以用小模型，复杂推理才用大模型。

多模型调度的核心是根据任务复杂度、延迟预算和成本目标选择合适模型。

小模型适合做意图识别、分类、抽取和简单改写，大模型更适合复杂生成和多步推理。

如果一个系统中既有小模型又有大模型，通常需要一个路由层决定调用顺序和切换条件。

模型 API 的熔断机制可以防止下游服务异常时把整个系统拖垮。

备用模型切换通常用于主模型限流、故障或响应过慢时，保证用户请求不中断。

Token 成本控制是大模型工程中的重要问题，因为上下文越长、输出越多，成本就越高。

限流、缓存、上下文裁剪和批量推理都可以帮助控制 Token 消耗。

长上下文不等于高质量，过长上下文反而可能引入噪声并增加推理成本。

在流式输出场景中，后端需要按片段返回结果，同时保持会话状态和序列顺序。

如果模型输出速度不稳定，前端可以通过缓冲和节流来减少渲染卡顿。

异步任务适合执行时间较长、需要多步处理或依赖外部系统的请求。

异步任务通常要配合状态机、消息队列或任务表来管理执行进度。

可观测性对大模型工程非常重要，因为很多问题不是“模型不会”，而是“系统哪一步出了问题”。

完整日志应记录输入、输出、模型版本、参数配置、工具调用和耗时信息。

Trace ID 可以把一次请求在多个服务之间串起来，方便排查跨服务问题。

在微服务架构里，AI 模块往往需要和 Java 或 Go 后端通过 HTTP、RPC 或消息队列进行交互。

如果 AI 模块涉及跨服务操作，最终一致性通常比强一致性更现实。

缓存可以分别作用于模型输出、检索结果、embedding 和中间工具结果。

缓存时间的设置要结合数据更新频率，更新快的内容应采用较短 TTL。

如果知识库经常更新，缓存策略要避免旧结果长期存在。

高并发下的队列调度一般要考虑优先级、超时、重试和拒绝策略。

资源隔离可以避免某类请求把全部 GPU 或 CPU 资源占满。

在容器化部署中，健康检查很重要，它能帮助平台发现服务是否还能正常响应。

滚动更新可以减少模型服务升级时的不可用窗口。

MLOps 在大模型工程里主要体现在模型版本管理、实验记录、配置管理和自动化部署上。

如果没有版本管理，模型更新后很难定位问题来自模型本身还是配置变化。

Prompt 不稳定时，最常见的优化办法是增加约束、提供示例和缩小任务范围。

在需要固定格式输出的场景中，结构化输出比纯自然语言更容易落地。

系统提示词应该尽量简洁、明确、稳定，不适合频繁改动。

用户提示词变化较大时，可以通过模板化来提高一致性。

如果任务包含多步推理，最好先让模型输出中间结果再继续下一步处理。

长文本生成常见问题是前后不一致、信息遗漏和结构混乱。

解决长文本生成问题时，常常会先生成大纲，再逐段展开。

如果目标是文档类长输出，分段生成和最终整合通常比一次性长输出更稳定。

评估大模型效果时，不能只看“回答像不像人话”，还要看准确率、相关性、稳定性和成本。

在知识增强任务里，模型是否正确利用了检索内容也是重要指标。

如果输出经常偏离问题，可能不是模型能力问题，而是 prompt、上下文或路由策略有问题。

模型服务的健康状态不仅看是否返回 200，还要看返回速度、错误率和资源占用。

大模型工程中经常需要做灰度发布，以便在小流量上验证新模型或新参数。

如果线上效果波动大，通常需要把模型版本、prompt 版本和检索版本一起管理。

大模型服务通常不是单点系统，而是“模型 + 检索 + 缓存 + 网关 + 日志”的组合。

如果系统强调稳定性，就要尽量把不可控因素减少，比如限制工具调用范围和输出格式。

对于工程岗位来说，真正重要的是“模型如何稳定服务业务”，而不是“模型内部每个数学细节”。

问题：什么是大模型工程师？ 答案：大模型工程师主要负责把大模型能力落地到实际系统中，工作内容通常包括模型选型、推理部署、接口封装、提示词设计、流式输出、性能优化和成本控制。重点不是训练模型本身，而是让模型稳定、高效地服务业务。

问题：大模型工程师和算法工程师有什么区别？ 答案：算法工程师更关注模型训练、优化方法、指标提升和实验分析；大模型工程师更关注模型如何上线、如何推理加速、如何控制成本、如何和业务系统集成。前者偏研究和建模，后者偏工程化和落地。

问题：推理服务和训练服务有什么区别？ 答案：训练服务主要关注梯度计算、训练效率和收敛效果；推理服务主要关注延迟、吞吐、稳定性和资源利用率。两者的系统设计目标完全不同，所以工程实现也不同。

问题：什么是模型预热？ 答案：模型预热是指在正式接流量前，先加载模型权重、初始化缓存或执行一次空推理，减少首个请求的冷启动延迟。它在在线推理场景中很常见。

问题：为什么在线推理要关注首 token 延迟？ 答案：首 token 延迟决定了用户感知到系统“开始响应”的速度。对于流式输出场景来说，首 token 越快，用户体验通常越好。

问题：什么是连续批处理？ 答案：连续批处理是把多个时刻到达的请求动态合并成批次一起推理，以提高 GPU 利用率和吞吐。它比固定批处理更适合在线推理场景。

问题：vLLM 为什么适合在线推理？ 答案：vLLM 在 KV Cache 管理和请求调度上做了优化，适合处理高并发的在线推理请求。它的目标是提高吞吐并降低显存压力。

问题：TGI 主要适合什么场景？ 答案：TGI 更偏向开箱即用的推理服务部署，适合快速搭建生产环境中的大模型 API 服务。它在工程集成和部署便利性上比较强。

问题：TensorRT-LLM 的特点是什么？ 答案：TensorRT-LLM 更强调极致性能优化，通常在支持的硬件和模型条件下，能提供更高的推理效率。它更适合对延迟和吞吐要求很高的场景。

问题：什么是模型量化？ 答案：模型量化是用更低精度的数据类型表示模型权重或激活值，以减少显存占用和加速推理。常见量化方式包括 INT8、INT4 等。

问题：INT8 和 INT4 量化有什么区别？ 答案：INT8 通常在压缩和精度之间较平衡，INT4 压缩更强、速度更快，但可能带来更明显的精度损失。具体选哪种要看业务对效果和性能的要求。

问题：量化后可能遇到哪些问题？ 答案：常见问题包括模型精度下降、输出不稳定、长文本效果变差和某些任务能力退化。上线前通常需要在真实任务上做回归测试。

问题：什么是多模型调度？ 答案：多模型调度是根据任务复杂度、成本和时延要求，动态选择不同模型处理请求。比如简单任务走小模型，复杂任务走大模型。

问题：为什么一个系统里常常要同时用小模型和大模型？ 答案：因为不是所有任务都值得用大模型处理。小模型适合分类、抽取、改写等轻任务，大模型适合复杂推理和生成，这样能兼顾成本和效果。

问题：什么是模型路由？ 答案：模型路由就是根据输入内容或任务类型，把请求分配给最合适的模型。它通常是多模型系统中的关键调度层。

问题：如何控制 Token 成本？ 答案：常见方式包括缩短上下文、使用缓存、选择合适模型、限制输出长度和采用批量推理。Token 控制是大模型工程里非常重要的一环。

问题：为什么长上下文不一定更好？ 答案：因为长上下文会增加推理成本，也可能引入噪声，干扰模型判断。真正有效的是把最相关的信息放进去，而不是无脑堆长文本。

问题：什么是流式输出？ 答案：流式输出是模型在生成过程中逐步返回内容，而不是等全部生成完成后一次性返回。它能显著提升用户感知速度和交互体验。

问题：流式输出对后端有什么要求？ 答案：后端需要支持分段返回、会话状态管理、错误处理和终止控制，同时保证内容顺序和连接稳定。流式实现通常比普通一次性返回更复杂。

问题：什么是异步任务？ 答案：异步任务适合执行时间较长或依赖外部系统的请求，提交后不会阻塞主线程，而是后台执行并在完成后通知结果。它常用于复杂 Agent 任务和长耗时推理。

问题：为什么大模型系统要做可观测性？ 答案：因为大模型服务的问题往往出在多个环节，比如模型本身、提示词、检索、工具调用或网络。可观测性可以帮助快速定位问题来源。

问题：日志里应该记录哪些内容？ 答案：通常要记录输入、输出、模型版本、参数配置、工具调用、耗时、错误信息和 trace ID。这样才能支持后续排障和分析。

问题：什么是 Trace ID？ 答案：Trace ID 是一次请求在多个服务之间的唯一追踪标识。它可以把网关、业务服务、模型服务和工具调用串联起来，方便全链路排查。

问题：大模型服务为什么要做熔断？ 答案：熔断可以在下游服务异常时及时切断请求，防止故障扩散和资源继续浪费。它是高可用系统中很常见的保护机制。

问题：备用模型有什么作用？ 答案：备用模型可以在主模型限流、超时或故障时接管请求，保证服务不中断。它是降级和灾备的一部分。

问题：什么是模型灰度发布？ 答案：灰度发布是先让一小部分流量使用新模型或新配置，观察效果稳定后再逐步放量。它能降低上线风险。

问题：为什么 Prompt 设计要尽量稳定？ 答案：因为 Prompt 的变化会直接影响模型输出风格和行为。稳定的 Prompt 更利于回归测试、版本管理和线上维护。

问题：什么是结构化输出？ 答案：结构化输出是让模型按固定字段或格式输出结果，比如 JSON、表格或标准列表。它比自由文本更容易被程序解析和校验。

问题：为什么工程场景常要求模型输出 JSON？ 答案：因为 JSON 易于程序读取、存储和展示，也便于后续做自动化处理。对于抽取、评分、分类和题目生成等任务特别有用。

问题：什么是 Prompt 中的 few-shot 示例？ 答案：few-shot 示例是在提示词中给出少量输入输出样例，帮助模型模仿任务格式和风格。它常用于抽取、分类和格式转换任务。

问题：什么是上下文裁剪？ 答案：上下文裁剪是只保留最相关的历史消息、检索片段或任务信息，去掉无关内容，以降低 token 消耗和噪声。

问题：大模型和 RAG 为什么常常一起用？ 答案：因为大模型负责生成和推理，RAG 负责提供外部知识支撑。两者结合后，模型回答通常更准确，也更有依据。

问题：如果检索结果不相关，生成结果会怎样？ 答案：模型很可能基于错误上下文继续生成，导致回答偏题或出现幻觉。所以 RAG 中检索质量往往直接影响最终输出。

问题：为什么需要重排 rerank？ 答案：因为初步召回的结果可能包含很多噪声，重排可以帮助从候选中挑出最相关的内容，提高上下文质量。

问题：什么是缓存？ 答案：缓存是把常用结果临时保存起来，下次直接读取以减少重复计算。大模型系统里常用于模型结果、检索结果和中间步骤结果。

问题：为什么要缓存检索结果？ 答案：因为很多用户会重复问相似问题，缓存能显著减少向量检索和重排的开销，提高响应速度。

问题：什么是负载均衡？ 答案：负载均衡是把请求分配到多个服务实例上，避免某一个实例压力过大。它是高并发系统的基础能力。

问题：什么是资源隔离？ 答案：资源隔离是让不同任务或不同服务使用独立的资源配额，避免互相抢占。对 GPU 资源紧张的大模型服务尤其重要。

问题：为什么大模型系统要做限流？ 答案：限流可以防止突发流量把模型服务打垮，也可以控制 Token 成本和资源消耗。它是保护系统稳定性的常用手段。

问题：大模型系统如何做降级？ 答案：当主模型或主服务异常时，可以切换到备用模型、缩短输出、减少上下文或改走模板化回答。降级的目标是保证核心服务可用。

问题：什么是长文本生成的分段策略？ 答案：分段策略是先生成大纲或分章节内容，再逐段扩展和整合。它比一次性生成长文本更稳，前后也更容易保持一致。

问题：为什么说大模型工程师要关注成本而不只是效果？ 答案：因为真实业务里，效果再好如果成本太高、延迟太大，也不一定能上线。工程落地必须在效果、成本和稳定性之间平衡。

问题：线上大模型服务最常见的性能指标有哪些？ 答案：常见指标包括首 token 延迟、总响应时间、吞吐、错误率、GPU 利用率和 Token 成本。它们共同反映服务质量。

问题：什么是灰度回滚？ 答案：灰度回滚是当新模型或新配置出现问题时，快速切回旧版本。它是线上发布安全性的关键机制之一。

问题：为什么要把模型版本和 Prompt 版本一起管理？ 答案：因为输出变化可能来自模型，也可能来自 Prompt。一起管理才能在问题出现时准确定位到底是哪一层变化导致的。

问题：什么是工程化落地的大模型能力？ 答案：工程化落地指不是停留在“模型能回答”，而是能稳定、可控、可监控、可扩展地提供服务。这个能力才真正符合生产要求。

问题：面试中如果问你大模型工程师最核心的能力是什么，你怎么答？ 答案：可以回答为：核心是把模型能力转化成稳定可用的线上服务，涵盖模型选型、推理加速、提示词设计、检索增强、流式输出、成本控制和可观测性等方面。
