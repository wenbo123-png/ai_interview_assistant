Agent 是一种能够根据任务自主决定是否调用工具、如何分解步骤并执行任务的智能体，适合处理多步骤、需要外部信息或需要动作执行的场景。

Tool Calling 是让大模型在需要时调用外部函数、API 或检索器来完成任务的机制，通常通过结构化参数传递给工具。

Function Calling 是一种标准化的工具调用方式，模型输出函数名和参数，由程序执行真实函数后再把结果返回给模型。

ReAct 是 Reason + Act 的缩写，指模型在推理和行动之间交替进行，先思考再调用工具，得到结果后继续推理。

Prompt Engineering 是通过设计输入提示词来控制模型输出行为、格式、风格和任务执行方式的技术。

RAG 是 Retrieval-Augmented Generation 的缩写，表示先从外部知识源检索相关内容，再结合检索结果生成答案。

Memory 指模型或 Agent 对历史上下文、用户偏好、任务状态等信息的保存与使用能力。

Planning 是 Agent 在执行任务前先进行步骤拆解和路径规划的能力，用于决定先做什么、后做什么。

LangChain 是一个用于快速构建大模型应用的开发框架，提供模型调用、提示词、检索、工具调用和 Agent 组合能力。

LangGraph 是一个用于构建复杂 Agent 工作流的框架，支持状态流转、分支、循环和图结构编排。

Structured Output 指让模型按照固定格式输出结果，例如 JSON、表格或指定字段，便于程序解析和后续处理。

Hallucination 是大模型生成看似合理但实际错误、编造或不可靠内容的现象，是 LLM 应用中的常见问题。

Retrieval 是从文档库、向量数据库或搜索系统中找出与查询最相关内容的过程，是 RAG 的核心环节。

Workflow 是一系列按顺序或按条件执行的步骤流程，用于把复杂任务拆成多个可控阶段。

Task Decomposition 是把一个复杂任务拆分成多个子任务的过程，常用于 Agent 规划与多步执行。

Embedding 是把文本、图片等信息映射到向量空间的表示方法，使语义相似内容在向量空间中更接近。

Vector Database 是用于存储向量并支持相似度检索的数据库，常用于知识库问答和 RAG 系统。

Chunking 是将长文本切分成多个片段的过程，便于向量化、检索和上下文拼接。

Chunk Overlap 是相邻文本块之间保留部分重叠内容的策略，用来减少切分造成的信息断裂。

Top-k Retrieval 是从检索结果中返回最相关的前 k 条内容，是向量检索和搜索系统中的常见做法。

Similarity Search 是根据向量之间的相似度来查找相关内容的检索方法，常见于语义检索场景。

Cosine Similarity 是衡量两个向量方向相似程度的指标，常用于文本向量检索。

BM25 是一种经典的关键词检索排序算法，适合处理包含明确词项匹配的搜索任务。

Hybrid Retrieval 是把向量检索和关键词检索结合起来的检索方式，兼顾语义匹配和词项匹配。

Rerank 是对初步召回的检索结果进行二次排序的方法，通常用更强的模型提升最终相关性。

Query Rewrite 是对用户原始问题进行改写，使检索查询更清晰、更适合召回相关知识。

Context Packing 是把检索到的相关内容组织成适合输入给大模型的上下文结构，以提升回答质量。

Prompt Template 是预先设计好的提示词模板，用于稳定地控制模型在不同任务中的行为。

System Prompt 是对模型角色、目标、边界和行为规则的高优先级指令，通常用于约束模型输出。

User Prompt 是用户直接输入的问题或任务描述，是模型实际处理的主要请求内容。

Few-shot Prompting 是在提示词中加入少量示例，让模型模仿示例完成任务的方式。

Zero-shot Prompting 是不给示例，直接让模型根据指令完成任务的方法。

Chain-of-Thought 是一种鼓励模型进行逐步推理的提示方法，用于提升复杂任务的推理质量。

Self-Ask 是模型先把大问题拆成若干小问题，再逐步回答的策略，常用于复杂问答场景。

Tool Selection 是 Agent 根据当前任务判断应该调用哪个工具的过程，是工具型 Agent 的关键能力。

Tool Schema 是对工具输入参数、输出结构和约束的定义，便于模型正确调用工具。

Retriever 是 RAG 中负责从知识库中找回相关内容的组件，通常基于向量相似度或关键词匹配。

Retriever-Augmented Generation 是在生成前先检索外部信息的生成范式，能降低幻觉并增强知识覆盖。

Document Loader 是读取 PDF、TXT、DOCX、网页等文档内容的组件。

Text Splitter 是把文档切分成适合检索和向量化的文本片段的工具。

Metadata 是附加在文档片段上的结构化信息，如来源、页码、标题、标签等。

Source Attribution 是在回答中标注信息来源的能力，能提高可解释性和可信度。

Answer Grounding 是让模型答案尽量基于检索到的真实资料，而不是凭空生成。

Context Window 是模型一次能处理的最大上下文长度，限制了输入文本和历史对话的规模。

Token 是模型处理文本时的基本单位，通常与字、词或子词相关。

Tokenizer 是把文本切分成 token 的工具，不同模型的分词方式可能不同。

Latency 是系统响应所需时间，是 Agent 和 RAG 系统的重要性能指标。

Throughput 是单位时间内系统可处理的请求数量，常用于衡量服务并发能力。

Error Handling 是系统在调用工具、检索知识或模型推理失败时进行异常处理和兜底的机制。

Fallback Strategy 是主方案失败后启用的备用方案，例如检索失败时直接给出保守回答。

Confidence Score 是模型或系统对答案可靠性的估计分数，用于判断是否需要进一步检索或拒答。

Prompt Injection 是用户试图通过恶意输入诱导模型违反原始指令或泄露系统提示词的攻击方式。

Guardrails 是对模型输出和行为的安全约束机制，用于防止越权、违规或不安全回答。

Agent 的任务流程通常包括理解用户意图、判断是否需要工具、选择合适工具、执行工具调用、整合结果并输出最终答案。

Agent 先做任务分解，是因为复杂任务往往不能一步完成，拆成多个子任务后更容易控制流程和结果质量。

工具调用和直接生成答案的区别在于，直接生成依赖模型参数记忆，而工具调用会把问题交给外部系统处理，准确性和实时性通常更高。

如果问题涉及实时信息、私有数据或精确计算，优先使用工具调用而不是让模型直接编造答案。

Agent 选择工具时，通常会先根据用户意图分类，再结合工具描述、输入参数和当前上下文决定调用哪个工具。

一个好的工具应该职责单一、输入清晰、输出稳定，这样 Agent 才容易学会正确使用它。

工具的输入输出最好结构化，尤其是输入参数要明确类型、字段名和含义，避免模型乱传参数。

在设计 Agent 工具时，最常见的做法是把复杂能力拆成多个小工具，而不是做一个什么都能干的大工具。

ReAct 的核心在于推理和行动交替进行，而不是先一次性想完所有步骤。

ReAct 流程中，模型先观察当前问题，再决定行动，拿到工具返回结果后继续下一步推理。

Prompt 的作用不只是让模型回答问题，还包括约束角色、规定格式、控制风格和指导思考路径。

系统提示词通常用来定义 Agent 的身份、目标、边界和安全规则，是整个应用行为约束的基础。

用户提示词应该尽量具体，因为任务描述越清楚，模型越容易稳定输出。

如果希望模型输出可被程序处理的结果，最好在 prompt 中明确要求结构化格式，比如固定字段或 JSON。

Few-shot 示例能显著提升模型在特定任务中的稳定性，尤其适合分类、抽取和格式化输出任务。

RAG 的基本流程是先检索，再生成，核心目标是让答案尽量建立在真实资料基础上。

RAG 系统里，检索质量往往比生成模型本身更重要，因为“找对资料”通常比“写得漂亮”更关键。

Chunk 切分策略会直接影响检索效果，切得太大容易噪声多，切得太小又容易丢失上下文。

Chunk overlap 的作用是保留相邻片段的上下文连续性，减少信息被切断的问题。

向量检索前要先做 embedding，因为只有把文本转成向量，系统才能进行语义相似度计算。

Embedding 模型的好坏会直接影响召回效果，不同领域的文本适合不同的 embedding 模型。

BM25 更适合关键词明确、术语敏感的任务，向量检索更适合语义相近但词面不同的任务。

混合检索可以同时兼顾关键词精确匹配和语义召回，适合技术面试这种既看术语又看语义的场景。

Rerank 通常放在初步召回之后，用来对候选结果重新排序，提升最终返回内容的相关性。

Query rewrite 的目的是把用户模糊的问题改写得更适合检索，尤其适合长问题、口语化问题和上下文不完整的问题。

Context packing 的重点不是把所有检索结果都塞进去，而是筛选最有用的信息并按逻辑组织好。

Document loader 的任务是把不同格式的文件统一转成可处理的文本内容。

如果文档中有标题、段落、页码等结构，尽量在加载阶段保留下来，后面做来源引用时会更方便。

Metadata 在 RAG 中很重要，因为它不仅能帮助过滤检索结果，还能帮助定位来源和增强可解释性。

Source attribution 能让回答更可信，尤其是在知识库问答或面试助手里非常有价值。

Hallucination 往往会在模型缺少上下文、检索失败或 prompt 约束不足时更容易出现。

减少幻觉的常见方法包括加强检索、限定回答范围、要求引用来源和对不确定内容保持保守。

Context window 不够时，系统需要做截断、摘要、分段检索或上下文压缩，否则模型容易丢信息。

Token 数量会影响调用成本和响应速度，因此在设计 prompt 和上下文时要控制长度。

Latency 过高时，用户体验会明显下降，所以 Agent 系统要尽量减少无效调用和冗余推理。

如果多个工具都能完成同一任务，最好让 Agent 优先选择最简单、最稳定、最便宜的那个。

Error handling 不能省略，因为模型调用、检索、文件读取和接口请求都可能失败。

Fallback strategy 的一个常见做法是：检索失败时给出基于当前上下文的保守回答，并明确提示信息不足。

Confidence score 可以帮助系统决定是否继续检索、是否需要追问用户，或者是否应该拒答。

Prompt injection 防护的重点是不要盲目相信用户输入，要把系统规则和用户内容分层处理。

Guardrails 不只是安全问题，也包括格式约束、内容边界和业务规则约束。

Task decomposition 常常先按“先分析、再检索、再生成、再校验”的顺序拆解。

复杂任务中，先规划再执行通常比边做边想更稳定，尤其适合多步骤 Agent。

如果一个问题需要多个知识点共同回答，先做子问题拆解通常能提升最终答案质量。

Memory 一般分为短期和长期两类，短期记住当前对话，长期记住跨会话信息或用户偏好。

短期记忆适合面试助手中的上下文连续追问，长期记忆更适合个性化建议和历史复盘。

在面试助手中，用户简历和 JD 更适合作为临时上下文，而不是长期记忆写进知识库。

Workflow 本质上是把复杂过程变成固定步骤或条件分支，适合做稳定的业务流程控制。

LangChain 更偏向组件拼装，适合快速搭建应用；LangGraph 更适合做有状态、有分支的复杂流程。

如果任务流程比较简单，LangChain 的 Agent 已经够用，不一定一开始就上 LangGraph。

Structured Output 可以显著降低后处理成本，因为程序可以直接读取字段而不是解析自然语言。

在抽取类任务中，要求模型输出固定 JSON 格式通常比纯文本回答更稳定。

如果模型经常输出格式错误，可以通过更严格的 prompt、示例和后处理校验来改善。

多轮对话中，上一轮回答会影响下一轮提问，所以上下文管理非常重要。

如果上下文太长，不一定要全塞进模型，可以只保留关键摘要和最近几轮最相关内容。

在知识库问答中，答案生成前最好先确认检索内容是否足够相关，否则容易胡编。

如果检索到的结果分散，可以先让模型做一次摘要整合，再进入最终回答阶段。

Agent 应用里，真正难的往往不是“让模型回答”，而是“让它按正确流程去做事”。

模型能力再强，如果工具设计混乱、prompt 不清晰、上下文组织差，最终效果也会很不稳定。

Agent 在实际应用中通常不是一次性完成全部任务，而是通过多轮决策逐步推进，因此它更像一个“会执行的工作流控制器”。

Agent 的状态管理很重要，因为它要知道当前任务做到哪一步、已经调用过哪些工具、还缺哪些信息。

状态设计通常包括用户输入、工具结果、当前步骤、历史对话和最终输出等字段。

如果 Agent 没有清晰的状态记录，就容易重复调用工具、遗漏步骤或者在多轮任务中迷失。

Tool Calling 的本质是把“让模型自己回答”改成“让模型先决定是否需要外部能力”。

工具调用适合做确定性任务，比如检索、计算、数据库查询、文件读取和结构化抽取。

直接生成答案适合做开放式表达、经验总结、自然语言解释，但不适合完全依赖事实准确性的场景。

在面试助手里，简历分析、JD 解析、题目生成、评分和追问都可以拆成不同工具，便于控制和维护。

一个工具返回的结果如果太长，最好先做摘要或提炼，否则会占用太多上下文窗口。

工具之间的依赖关系要提前设计好，例如简历解析结果往往是面试题生成和评分的前置输入。

ReAct 里的“Act”不只是调用工具，也包括决定是否继续循环、是否结束任务、是否追问用户。

ReAct 的优势在于过程透明，便于调试和排查 Agent 为什么做出某个判断。

Prompt Engineering 的核心不是堆字，而是把任务目标、约束条件、输出格式和示例组织清楚。

对于抽取类任务，prompt 里最好明确“只输出结果，不要解释过程”，这样更容易后处理。

如果任务要求模型按照岗位需求生成内容，prompt 中需要给出岗位目标、内容边界和输出结构。

RAG 的检索结果如果和问题不够相关，模型即使再强也很难生成高质量答案。

检索增强的价值不仅在于“补知识”，还在于“缩小回答范围”，让模型聚焦在可靠信息上。

在知识库构建阶段，文档质量比数量更重要，重复、无关和噪声数据会拉低整体效果。

文本切分时要尽量按语义边界切，而不是机械按固定长度切，否则检索片段容易割裂。

如果文档里有标题层级，切分时最好把标题保留在片段里，方便模型理解上下文。

向量数据库里常见的操作不仅是新增和查询，还包括删除、更新和按元数据过滤。

如果知识库内容经常更新，就要考虑增量入库和版本管理，避免旧数据和新数据混在一起。

Embedding 不同于文本生成，embedding 更关注语义表示，输出的是向量而不是自然语言。

如果领域词汇很多，比如 AI、算法、NLP 术语，最好准备更贴近该领域的知识文本来做向量检索。

BM25 对精确词匹配更敏感，特别适合技能名、论文名、技术栈这类明确关键词。

Hybrid Retrieval 常常先做双路召回，再合并结果，最后交给 rerank 模型筛选。

Rerank 的输入通常是“用户问题 + 候选文档片段”，它的目标是判断哪个片段真正更相关。

Query Rewrite 常用于把“口语化问法”转成“更适合检索的标准问法”。

如果用户问法很模糊，可以先做 query rewrite，再检索，再回答，效果通常更好。

Context packing 要兼顾“相关性”和“紧凑性”，不是越多越好，而是越准越好。

如果上下文里存在冲突信息，最好优先保留最新、最可靠或者来源更明确的内容。

Structured Output 的一个常见用途是让模型输出题目列表、评分结果、技能点清单等固定字段。

在工程里，结构化输出往往比自然语言更适合存档、分析和展示。

如果模型输出经常不符合格式，可以在 prompt 里加入字段模板和字段说明来约束。

Memory 不是简单的“保存聊天记录”，而是要选择哪些信息值得保留、如何压缩、何时调用。

短期记忆通常只保留最近轮次，长期记忆可以保存用户背景、偏好和历史任务结论。

如果长期记忆太多，反而会污染当前任务，所以需要检索式记忆或摘要式记忆来控制规模。

Planning 可以是显式规划，也可以是隐式规划；显式规划更容易调试，隐式规划更自然但不稳定。

一个实用的 Agent 规划方式是先生成子任务清单，再逐步执行并在每一步校验结果。

Task Decomposition 不仅能降低复杂度，还能让不同步骤使用不同工具，提高整体准确率。

Workflow 和 Agent 的区别之一在于，Workflow 更偏固定流程，Agent 更偏动态决策。

如果业务流程变化不大，用 Workflow 往往比纯 Agent 更稳定、更容易测试。

LangChain 适合封装常见能力，比如模型调用、检索、链式组合和工具注册。

LangGraph 的优势是能把复杂任务表示成图结构，适合循环、条件分支和多状态管理。

如果系统需要“先判断，再分支，再回到上一步”的逻辑，LangGraph 会比简单链式调用更合适。

Prompt Injection 的防护可以通过分层提示、输入清洗、工具白名单和输出约束来增强。

Guardrails 不仅用于安全审查，也可以用于限制 Agent 的可回答范围和回答风格。

如果模型对某个问题没有足够依据，最好的做法不是硬答，而是明确说明信息不足。

在面试助手里，知识库检索失败时，可以退化成“基于简历和 JD 做经验性分析”的模式。

Tool Selection 受工具描述影响很大，所以工具命名一定要清晰、直观、避免歧义。

工具名称最好体现“做什么”，而不是抽象名词，这样模型更容易理解和调用。

一个好的 Agent 设计要尽量减少工具数量，但又要保证职责拆分清晰。

如果工具过多，模型会更难选择；如果工具过少，单个工具又会太复杂。

在实际应用中，最常见的错误之一是把“检索”和“生成”混在一个步骤里，导致难以调试。

更好的方式通常是先独立完成检索，再把检索结果交给生成模块。

如果一个任务需要高可靠性，最好在生成后再做一次校验，比如格式校验、事实校验或规则校验。

对于面试题生成，常见的流程是：解析简历和 JD → 找出关键词 → 检索知识库 → 生成题目 → 生成考察点。

对于答案评分，常见的流程是：提取用户回答要点 → 对照标准要点 → 按维度打分 → 输出改进建议。

对于追问生成，常见做法是针对回答中的“模糊点”“遗漏点”“风险点”继续发问。

如果用户回答过于简短，追问通常应该先引导展开，再逐步深入。

如果用户回答已经很完整，追问可以转向项目细节、工程实现或优化思路。

Agent 应用常常要做“保守输出”，也就是在不确定时少说、准说，而不是强行给满答案。

在知识库问答中，引用来源会提升可信度，也方便用户验证答案是否正确。

如果需要多轮对话连贯性，最好把当前任务目标显式存入会话状态，而不是只靠原始对话文本。

有些任务不需要 Agent 介入，直接用检索 + 生成链路就够了，因此不要为了“Agent”而强行 Agent 化。

对于实习项目来说，系统的可解释性、可演示性和稳定性，往往比复杂度更重要。

模型在 Agent 中的角色不只是“回答问题”，还包括“判断任务、决定工具、整合结果”和“控制输出边界”。

一个成熟的 Agent 应用通常会把“理解任务”和“执行任务”分开处理，这样更便于调试和扩展。

在任务开始时先做意图识别，可以帮助系统判断用户是在提问、要求生成、还是要求分析。

意图识别的结果会影响后续流程，比如是进入检索、生成、评分，还是进入多轮追问模式。

如果用户输入中包含多个子目标，最好先做任务拆解，再逐个处理，而不是一次性全部交给模型。

任务拆解的好处是能够降低模型一次性处理的负担，也能让每一步的结果更可控。

在面试助手中，用户上传简历后，系统可以先抽取技能、项目和经历，再据此生成不同类型的题目。

岗位 JD 解析的重点不是逐字理解，而是抽取出职责、技能要求、加分项和隐含能力需求。

隐含需求往往体现在岗位描述中反复出现的关键词，比如“沟通能力”“落地经验”“工程化”等。

如果一个任务同时要求事实准确和表达自然，通常先保证事实，再优化表达。

大模型在事实抽取任务中容易漏字段，所以结构化抽取比开放式长文本回答更可靠。

抽取结果最好做一次字段校验，确保没有关键字段缺失或格式错误。

RAG 系统中的检索质量不仅和 embedding 有关，也和文档切分、索引构建、查询表达有关。

如果检索结果经常不相关，优先检查数据源、切分方式和查询改写，而不只是换模型。

在构建知识库时，应该尽量保证同一主题下的文档风格一致，这样检索后的拼接效果更稳定。

向量数据库适合做语义检索，但不适合承载所有业务逻辑，业务判断应该放在 Agent 或应用层。

如果知识库内容太杂，检索结果容易混乱，因此建议按照岗位方向或主题分类存放。

不同岗位的知识库可以共享一部分基础内容，比如 Transformer、RAG、Prompt Engineering 等。

但对于不同岗位的专属内容，比如 Agent 工程、算法工程、NLP 工程，最好分别组织专题文档。

Prompt 设计中，任务说明越明确，模型越容易输出符合预期的结果。

如果模型总是输出多余解释，可以在提示词中明确要求“只输出结果，不要解释过程”。

如果模型总是漏掉重点，可以在 prompt 中加入“必须覆盖以下要点”的约束。

在多轮任务中，上一轮结果往往要作为下一轮输入的一部分，否则 Agent 很容易丢失上下文。

一个常见做法是把上一轮输出压缩成摘要，再作为后续步骤的输入。

摘要式上下文压缩可以减少 token 消耗，也能降低上下文噪声。

Memory 的构建需要考虑“保留什么、删除什么、压缩什么”，而不是无限堆积。

长期记忆不适合保存太多原始对话，通常更适合保存结论、偏好和重要事件。

如果用户的目标会变化，Agent 应该允许更新任务状态，而不是死记最初目标。

Planning 的价值在于让 Agent 先决定路径，再开始执行，而不是边做边乱跳。

在复杂任务里，规划阶段可以先输出步骤列表，再逐步执行并记录结果。

对于自动化面试助手，规划的一个典型顺序是：分析简历 → 分析 JD → 找知识点 → 出题 → 评分 → 追问。

Task Decomposition 有助于把一个大任务变成多个可测试的小任务，提高工程可维护性。

如果每个子任务都能单独验证，那么整个系统的调试效率会高很多。

Workflow 适合做相对固定的流程，比如文档解析、入库、检索、生成、评分这类步骤链。

Agent 更适合处理流程中存在不确定性、分支判断和动态决策的部分。

LangChain 中常见的链式思路是“输入经过若干模块后得到输出”，适合简单和中等复杂度任务。

LangGraph 则更适合把任务做成状态机图，能明确表达跳转、循环和条件分支。

如果后面想增加“用户不满意则重新追问”“检索不到则改写问题再检索”这种逻辑，LangGraph 会更有优势。

Structured Output 不仅方便程序解析，也方便后续做统计、展示和存档。

在面试题生成场景中，结构化输出可以包括题目、考察点、难度、建议回答方向等字段。

在评分场景中，结构化输出可以包括总分、分项分数、优点、缺点和改进建议。

Hallucination 的本质不是“模型故意说假话”，而是模型在信息不足时仍然尝试补全答案。

降低幻觉的关键不是一味限制模型，而是给它足够可信的上下文和明确边界。

有时候模型表现不好，不是模型本身差，而是输入上下文没有整理好。

Retrieval 的本质是“从大量候选中找出最可能相关的少量内容”，所以召回和排序都很重要。

如果召回数量太少，容易漏掉关键内容；如果召回太多，生成阶段会被噪声干扰。

Top-k 的设置需要根据任务调整，既不能太少，也不能太多。

在工程里，知识库检索常常不是一次完成，而是“先召回、再重排、再筛选”。

Embedding 模型的输入最好是清洗过的文本，去掉明显噪声、重复项和无意义字符。

如果文档里存在标题、列表、表格，最好在处理时尽量保持结构信息，而不是粗暴扁平化。

对于岗位相关知识库，标题命名要清晰，比如“Agent 基础”“RAG 检索”“大模型工程基础”等，方便检索和维护。

面试高频题库不仅要有问题，还要有“标准考点”和“参考答题方向”。

如果只收集问题而没有知识解释，知识库对用户的帮助会比较有限。

在知识库中，概念解释、实现流程、常见坑和面试题最好都保留，这样更利于 RAG 回答。

如果某个知识点很重要，最好同时准备“定义版”“原理版”“实现版”“面试版”四种材料。

Agent 工程中最常见的挑战之一，是让模型在“自由表达”和“严格执行”之间保持平衡。

如果太自由，输出容易不稳定；如果太严格，又可能失去自然语言能力。

一个实用的方法是让模型在内部按步骤处理，但对外只输出简洁结果。

如果你后面要把这些知识放进 txt 文件，建议每条内容尽量保持一句到三句，便于检索和切分。

对知识库来说，短条目更容易被向量化检索命中，长段落则更适合做参考阅读。

不同知识点之间最好保持语义独立，避免一条里塞太多概念，影响后续召回精度。

如果后续要做更高级的检索，可以考虑把知识点按“概念、原理、实现、面试题”四个维度标注。

在应用层，用户更关心“怎么用”和“怎么答”，所以知识库内容应该尽量面向实际任务。

对于 AI 面试助手来说，知识库不是论文集合，而是“可检索的面试知识与答题素材库”。

问题：什么是 AI Agent？ 答案：AI Agent 可以理解为一种能够根据目标自主思考、选择工具并执行任务的智能体。它不只是回答用户问题，还会判断当前任务需不需要检索、计算、调用接口或分步骤执行，然后把结果整合后输出。

问题：AI Agent 和普通 Chatbot 有什么区别？ 答案：普通 Chatbot 通常是“用户问一句，它答一句”，主要依赖模型直接生成回复。AI Agent 则更强调任务执行能力，它会先分析任务，再决定是否调用工具、是否需要多步处理，因此更适合复杂流程和真实业务场景。

问题：为什么大模型系统里需要工具调用？ 答案：因为大模型本身并不总是掌握实时信息，也不适合直接处理所有确定性任务。工具调用可以让模型去查数据库、检索知识、做计算、读文件或访问外部服务，从而提高准确性和可控性。

问题：什么是 Function Calling？ 答案：Function Calling 是一种让模型输出函数名和参数，由程序去执行真实函数的机制。模型本身不直接完成动作，而是通过结构化方式请求调用工具，执行结果再反馈给模型继续处理。

问题：什么是 ReAct？ 答案：ReAct 是 Reason + Act 的缩写，表示模型在推理和行动之间交替进行。它先思考当前问题，再决定调用哪个工具，拿到工具结果后继续推理，最后给出最终答案。

问题：Agent 的典型任务流程是什么？ 答案：通常是先理解用户意图，再判断是否需要工具，然后选择合适工具并执行调用，接着整合工具结果，最后生成最终回答。如果任务比较复杂，还可能在中间反复多次调用工具。

问题：为什么 Agent 不直接让模型一次性生成答案？ 答案：因为很多任务需要外部信息、精确计算或多步处理，直接生成容易出错。通过先调用工具再生成，系统能更准确、更稳定，也更容易调试和控制。

问题：什么是 Prompt Engineering？ 答案：Prompt Engineering 就是通过设计提示词来引导模型完成任务的方法。它包括角色设定、任务说明、输出格式约束、示例引导和边界控制等内容。

问题：为什么 Prompt 对 Agent 很重要？ 答案：因为 Agent 的行为很大程度上受 prompt 控制。好的 prompt 能让模型更清楚什么时候调用工具、输出什么格式、遇到不确定时怎么处理，从而提升稳定性。

问题：什么是 RAG？ 答案：RAG 是 Retrieval-Augmented Generation 的缩写，意思是在生成回答前先从外部知识源检索相关内容，再结合检索结果进行回答。它可以减少幻觉，并让回答更依赖真实资料。

问题：RAG 的核心流程是什么？ 答案：RAG 一般包括文档处理、切分、向量化、检索、上下文拼接和最终生成几个步骤。先把知识库内容准备好，再在用户提问时检索相关片段交给模型回答。

问题：为什么要做文本切分 Chunking？ 答案：因为长文档不适合直接整体向量化和检索，切分后能把内容拆成更容易命中的小片段。合理的切分还能兼顾上下文连续性和检索精度。

问题：什么是 chunk overlap？ 答案：chunk overlap 是相邻文本块之间保留部分重叠内容的策略。它的作用是减少文本切分造成的信息断裂，让前后片段之间保持一定上下文联系。

问题：什么是 Embedding？ 答案：Embedding 是把文本映射成向量表示的过程。向量之间的距离或相似度可以反映语义接近程度，因此适合用于语义检索和相似内容匹配。

问题：向量数据库的作用是什么？ 答案：向量数据库用于存储文本向量，并支持相似度检索。它可以帮助系统快速找到和用户问题语义最接近的知识片段，是 RAG 的基础设施之一。

问题：什么是 Similarity Search？ 答案：Similarity Search 是根据向量相似度来查找相关内容的检索方法。系统会把问题向量和库中文档向量做比较，返回最相近的内容。

问题：什么是 BM25？ 答案：BM25 是一种经典的关键词检索排序方法，特别适合词面匹配明确的查询。它对技术名词、岗位关键词、专有名词这类任务很有帮助。

问题：什么是混合检索？ 答案：混合检索是把向量检索和关键词检索结合起来的一种方法。它既能利用语义相似度召回相关内容，也能利用关键词精确匹配，提高检索效果。

问题：什么是 Rerank？ 答案：Rerank 是对初步召回的候选结果进行二次排序的过程。它通常会用更强的模型判断哪些片段真正和问题更相关，从而提升最终返回质量。

问题：什么是 Query Rewrite？ 答案：Query Rewrite 是把用户原始问题改写成更适合检索的形式。这样可以改善口语化表达、长问题或上下文不完整带来的检索效果。

问题：什么是 Structured Output？ 答案：Structured Output 指模型输出固定格式结果，比如 JSON、字段列表或表格。这样程序更容易读取和处理，也适合做系统化展示。

问题：为什么面试助手中常要求结构化输出？ 答案：因为面试助手不仅要“能说”，还要“能被程序处理”。比如生成题目、评分、追问和建议时，结构化输出更方便前端展示和后端存储。

问题：什么是 Memory？ 答案：Memory 指系统对历史上下文、用户偏好和任务状态的保存能力。它能让 Agent 在多轮交互中记住之前发生的事情，而不是每次都从头开始。

问题：短期记忆和长期记忆有什么区别？ 答案：短期记忆主要保存当前会话中的上下文信息，适合多轮对话连续处理。长期记忆则保存跨会话的用户偏好或历史结论，更适合个性化场景。

问题：什么是 Task Decomposition？ 答案：Task Decomposition 就是把一个复杂任务拆成多个小任务。这样可以降低单次处理难度，也能让每一步更容易验证和调试。

问题：什么是 Planning，在 Agent 里有什么作用？ 答案：Planning 指 Agent 在执行任务前先进行步骤规划。它会先决定任务怎么拆、先做什么、后做什么，再逐步执行，这样能提高复杂任务的稳定性和可控性。

问题：为什么 Agent 通常要先做任务拆解？ 答案：因为复杂任务往往不能一步完成，直接让模型一次性输出容易遗漏关键步骤。先拆解成多个子任务后，可以分别处理、分别校验，整体效果通常更好。

问题：什么是 Workflow，它和 Agent 有什么区别？ 答案：Workflow 是预先定义好的固定流程，通常按照顺序或条件分支执行。Agent 更强调动态决策，会根据当前情况决定下一步怎么做，所以更灵活，但也更不稳定。

问题：LangChain 的主要作用是什么？ 答案：LangChain 是一个用于快速构建大模型应用的框架。它提供了模型调用、提示词管理、检索、工具调用和 Agent 组合等能力，适合搭建原型和应用。

问题：LangGraph 适合解决什么问题？ 答案：LangGraph 更适合处理复杂、有状态、带分支和循环的任务流程。比如用户需要多轮追问、失败后重试、检索不到结果就改写查询，这类逻辑用 LangGraph 会更清晰。

问题：为什么有些场景不适合一开始就上 LangGraph？ 答案：因为如果任务本身比较简单，直接用 LangChain 或普通链式调用就够了。LangGraph 的设计更适合复杂流程，过早引入可能增加学习和维护成本。

问题：什么是 Hallucination？ 答案：Hallucination 指模型生成了看起来合理但实际上错误、编造或不可靠的内容。它是大模型应用中非常常见的问题，尤其在信息不足或没有检索支撑时更容易出现。

问题：如何减少模型幻觉？ 答案：常见方法包括使用 RAG 提供真实上下文、限制回答范围、要求引用来源、增加结果校验，以及在不确定时让模型保守回答而不是硬编。

问题：什么是 Context Window？ 答案：Context Window 是模型一次能够处理的上下文长度限制。输入内容、历史对话和检索结果都要算在这个范围内，所以设计系统时要控制上下文大小。

问题：为什么要控制 token 数量？ 答案：因为 token 数量会影响模型调用成本、响应速度和上下文容量。token 越多，通常越贵、越慢，而且更容易超出上下文限制。

问题：什么是 Source Attribution？ 答案：Source Attribution 指在回答中标明信息来源。它可以提高回答的可信度，也方便用户检查答案是否来自知识库中的真实内容。

问题：为什么知识库问答要保留来源信息？ 答案：因为这样更容易让用户判断回答是否可靠，也方便调试检索效果。如果答案出错，可以快速追溯是检索片段问题还是生成环节问题。

问题：什么是 Prompt Injection？ 答案：Prompt Injection 是用户通过特殊输入试图诱导模型忽略原始规则、泄露系统提示词或执行不该执行的行为。它是 Agent 和 LLM 系统里需要重点防范的风险。

问题：什么是 Guardrails？ 答案：Guardrails 是对模型行为的约束机制，用来限制它的输出范围、格式、安全性和业务边界。它可以防止模型乱答、越权或输出不符合要求的内容。

问题：为什么工具名称要设计得清晰？ 答案：因为模型会根据工具名称和描述决定是否调用它。名称越明确，模型越容易理解工具用途，也越不容易调用错误工具。

问题：一个好的工具应该具备什么特点？ 答案：一个好的工具通常职责单一、输入明确、输出稳定、失败行为可预测。这样 Agent 才更容易正确调用，也更容易维护和调试。

问题：为什么不建议把所有能力都塞进一个大工具？ 答案：因为大工具会让输入输出变复杂，模型也更难判断什么时候调用它。拆成多个小工具后，每个工具更专注、更稳定，也更容易复用。

问题：什么是 Tool Schema？ 答案：Tool Schema 是对工具参数和返回结果的结构化定义。它能帮助模型更准确地传参，也方便程序校验工具输入输出。

问题：为什么结构化的工具参数比自由文本更好？ 答案：因为结构化参数更容易被程序解析，也更不容易出现歧义。对于 Agent 来说，清晰的字段定义能显著减少调用错误。

问题：为什么用户上传内容不适合默认写入长期知识库？ 答案：因为用户上传的简历和 JD 只是当前任务相关内容，如果长期保存会导致数据膨胀、噪声增加，还可能带来隐私和检索质量问题。

问题：什么是知识库的固定数据和临时数据？ 答案：固定数据是提前构建好的通用知识，比如面试知识、技术资料和题库。临时数据是用户当前上传的简历、JD 和对话内容，通常只在当前会话中使用。

问题：什么是 Retrieval 在 RAG 里的核心意义？ 答案：Retrieval 的核心意义是先把相关资料找出来，再交给模型生成答案。这样可以让答案更有依据，也能覆盖模型参数里没有记住的信息。

问题：为什么说“找对资料”比“写得漂亮”更重要？ 答案：因为如果检索内容不相关，模型即使语言表达很好，答案也可能偏题或错误。RAG 系统的关键是先保证信息来源正确，再考虑表达质量。

问题：如果检索结果太多，会有什么问题？ 答案：如果召回结果太多，噪声会增加，模型容易被无关内容干扰，反而降低回答质量。所以通常需要筛选、重排或压缩上下文。

问题：如果检索结果太少，会有什么问题？ 答案：如果结果太少，系统可能漏掉关键知识，导致回答不完整或不准确。因此检索时要在“召回充分”和“上下文紧凑”之间做平衡。

问题：什么是多轮对话中的上下文管理？ 答案：上下文管理就是决定哪些历史信息要保留、哪些要压缩、哪些要丢弃。它能保证模型在多轮交互中始终围绕当前任务，不会乱掉。

问题：为什么上下文不能无限保存？ 答案：因为模型的上下文长度有限，而且过多历史内容会增加噪声和调用成本。通常只保留最相关的信息和必要摘要更合理。

问题：什么是模型输出校验？ 答案：模型输出校验是指在模型返回结果后，对格式、字段、内容完整性或规则进行检查。如果不符合要求，就重新生成或做后处理修正。

问题：为什么 Agent 系统要做兜底处理？ 答案：因为模型调用、工具调用和检索都可能失败。兜底处理可以避免系统直接崩掉，保证用户至少能拿到一个可用的回答或提示。

问题：什么是 Embedding，为什么它在 RAG 里很重要？ 答案：Embedding 是把文本转换成向量表示的过程，语义相近的文本在向量空间中也会更接近。在 RAG 中，检索依赖向量相似度，所以 embedding 的质量会直接影响召回效果。

问题：向量相似度在检索系统里通常怎么用？ 答案：向量相似度用于衡量用户问题和知识库片段之间的语义接近程度。系统会先把查询和文档都转成向量，再通过相似度排序找出最相关的内容。

问题：什么是 Cosine Similarity？ 答案：Cosine Similarity 用来衡量两个向量方向是否相近，数值越接近 1 表示越相似。在文本检索中，它是最常用的相似度指标之一。

问题：为什么文本切分不能太碎也不能太大？ 答案：切得太碎会让每个片段上下文不足，影响理解；切得太大又会引入过多噪声，降低检索精度。合理的切分要在上下文完整性和召回准确性之间做平衡。

问题：Chunk overlap 的作用是什么？ 答案：Chunk overlap 是在相邻文本块之间保留重叠内容，用来减少切分造成的信息断裂。它能让前后片段保持上下文连续性，提高检索和生成质量。

问题：什么是 Metadata，为什么它对检索有帮助？ 答案：Metadata 是附加在文档片段上的结构化信息，比如来源、标题、页码和标签。它能帮助过滤、排序和追踪来源，也方便后续引用和调试。

问题：为什么知识库文档要按主题分类？ 答案：按主题分类可以减少检索噪声，提高召回准确率，也方便后续维护。比如 Agent、RAG、LangChain、算法、NLP 分开组织，会更容易命中相关内容。

问题：什么是文本清洗？ 答案：文本清洗是去除文档中的多余空格、乱码、重复内容和无意义符号的过程。它能让后续切分、向量化和检索更稳定。

问题：Document Loader 的作用是什么？ 答案：Document Loader 负责把 PDF、TXT、DOCX 等不同格式的文件读取成统一文本。它是构建知识库的第一步。

问题：为什么在加载文档时要尽量保留结构信息？ 答案：因为标题、段落和页码等结构信息有助于后续切分、检索和来源定位。保留结构还能让模型更容易理解上下文。

问题：什么是 Query Understanding？ 答案：Query Understanding 是理解用户问题真实意图的过程，包括关键词、约束条件和上下文信息。它能帮助系统决定是否要改写问题、调用工具或直接检索。

问题：什么是 Query Rewrite？ 答案：Query Rewrite 是把用户原始问题改写成更适合检索的表达。它对口语化、信息不完整或者问题太长的情况特别有帮助。

问题：为什么复杂问题常常要先做问题改写？ 答案：因为用户原始问法可能不够标准，直接检索容易召回不准。先改写成更清晰的查询，通常能提高召回效果。

问题：什么是 Rerank？ 答案：Rerank 是对初步召回的候选结果进行二次排序的过程。它通常使用更强的模型来判断哪几个片段真正和问题最相关。

问题：为什么检索系统里常常要先召回再重排？ 答案：因为召回阶段追求广覆盖，可能会带来噪声；重排阶段追求高精度，能从候选中筛出最相关的内容。两步结合通常效果更好。

问题：什么是混合检索？ 答案：混合检索是把向量检索和关键词检索结合起来的方法。它既能利用语义匹配，也能利用关键词精确匹配，适合技术场景。

问题：BM25 和向量检索有什么区别？ 答案：BM25 更依赖关键词匹配，适合技术名词和明确术语；向量检索更关注语义相似，适合表达不同但意思相近的问题。两者各有优势。

问题：什么是 Context Packing？ 答案：Context Packing 是把检索到的相关内容整理成适合输入给模型的上下文。它的重点是保留最有用的信息，同时控制长度和噪声。

问题：为什么上下文不能无限堆叠？ 答案：因为模型的上下文长度有限，而且过多内容会增加噪声和调用成本。通常只保留最相关的片段和必要摘要更合理。

问题：什么是 Source Attribution？ 答案：Source Attribution 指在回答中标明信息来源。它可以增强回答可信度，也方便用户检查答案是从哪些资料中来的。

问题：为什么知识库问答要保留来源信息？ 答案：因为这能帮助判断答案是否可靠，也方便排查是检索问题还是生成问题。如果答案有误，可以快速定位来源片段。

问题：什么是 Hallucination？ 答案：Hallucination 指模型生成了看似合理但实际上错误或编造的内容。它是大模型应用中非常常见的问题，尤其在信息不足时更容易出现。

问题：怎么减少 Hallucination？ 答案：常见方法包括 RAG 检索增强、限制回答范围、加强 prompt 约束、增加结果校验，以及在不确定时保守回答。

问题：什么是 Prompt Injection？ 答案：Prompt Injection 是用户试图通过特殊输入干扰模型原始指令的攻击方式。它可能诱导模型忽略规则、泄露提示词或输出不安全内容。

问题：Guardrails 在 Agent 系统里有什么作用？ 答案：Guardrails 用于限制模型输出的范围、格式和安全边界。它可以防止模型乱答、越权或者违反业务规则。

问题：Tool Schema 为什么重要？ 答案：Tool Schema 定义了工具的输入输出格式和参数含义。它能帮助模型更准确地调用工具，也方便程序校验和调试。

问题：为什么工具的输入参数要尽量结构化？ 答案：结构化参数更容易被程序解析，也更不容易产生歧义。对于 Agent 来说，清晰的字段设计能显著减少调用错误。

问题：一个好的工具通常具备哪些特点？ 答案：一个好的工具通常职责单一、输入清晰、输出稳定、失败行为可预测。这样 Agent 才更容易正确调用，也更容易维护。

问题：为什么不建议把所有能力塞进一个大工具里？ 答案：因为大工具会让输入输出变复杂，模型也更难判断什么时候该调用它。拆成多个小工具后，每个工具更清晰、更稳定、更容易复用。

问题：什么是 Task Decomposition？ 答案：Task Decomposition 是把一个复杂任务拆成多个小任务的过程。这样可以降低一次性处理的难度，也更方便逐步校验结果。

问题：为什么复杂任务适合先拆解再执行？ 答案：因为拆解后每一步都更具体，模型更容易做对，系统也更容易调试。对于多步骤流程来说，这通常比一步生成更稳定。

问题：什么是 Planning？ 答案：Planning 指 Agent 在执行前先规划任务步骤。它会先决定先做什么、后做什么，再按顺序推进，从而提高复杂任务的可控性。

问题：Planning 和 Task Decomposition 有什么关系？ 答案：Task Decomposition 是把任务拆开，Planning 是决定这些子任务的执行顺序和策略。两者经常一起使用。

问题：什么是 Workflow？ 答案：Workflow 是一套预先定义好的执行流程，通常按照固定步骤或条件分支推进。它适合做稳定、可测试的业务流程控制。

问题：Workflow 和 Agent 有什么区别？ 答案：Workflow 偏固定流程，Agent 偏动态决策。Workflow 适合流程明确的任务，Agent 更适合需要判断和灵活调用工具的场景。

问题：LangChain 的优势是什么？ 答案：LangChain 提供了模型调用、提示词、检索、工具调用和链式组合等基础能力，开发效率高，适合快速搭建 LLM 应用。

问题：LangGraph 为什么适合复杂 Agent 流程？ 答案：因为 LangGraph 支持状态流转、分支和循环，能清楚表达“失败重试”“继续追问”“条件跳转”等逻辑，适合复杂工作流。

问题：什么是 Structured Output？ 答案：Structured Output 指让模型输出固定格式内容，比如 JSON、字段列表或表格。这样程序更容易解析，也更适合后续展示和存档。

问题：为什么在抽取类任务里常要求结构化输出？ 答案：因为抽取类任务需要程序直接使用结果，结构化格式比自然语言更稳定、更好处理，也更便于校验。

问题：什么是 Memory？ 答案：Memory 是系统保存历史上下文、任务状态或用户偏好的能力。它可以让 Agent 在多轮交互中维持连续性。

问题：短期记忆通常保存什么？ 答案：短期记忆通常保存当前会话中最相关的对话内容、当前任务状态和最近的工具调用结果，便于后续步骤继续处理。

问题：长期记忆通常适合保存什么？ 答案：长期记忆更适合保存用户偏好、历史任务结论或跨会话重要信息，不适合保存大量原始对话。

问题：为什么上下文管理在 Agent 中很重要？ 答案：因为 Agent 要在多轮推理和多次工具调用中保持任务连贯。如果上下文管理不好，就容易重复、跑偏或丢失关键信息。

问题：什么是可解释性？ 答案：可解释性是指系统为什么这么回答、为什么调用这个工具、为什么检索这些资料都能说清楚。它在 Agent 和 RAG 系统里非常重要。

问题：为什么 Agent 系统要强调可解释性？ 答案：因为 Agent 往往涉及多步决策和工具调用，用户需要知道系统是怎么得出结论的。可解释性越强，系统越容易被信任，也越方便排查问题。

问题：什么是保守回答？ 答案：保守回答是指在信息不足时不硬编、不夸大，而是明确说明依据有限，并给出当前能确定的部分。它能减少错误输出。

问题：为什么在知识不足时不能强行回答？ 答案：因为强行回答很容易引入幻觉，导致信息不准确。对面试助手来说，保守回答比瞎猜更可靠。

问题：什么是“检索优先”策略？ 答案：检索优先就是当问题涉及知识库、文档或事实信息时，优先先检索再回答，而不是直接生成。这样能提高答案准确性。

问题：如果检索结果很少，系统该怎么做？ 答案：可以尝试改写查询、放宽检索条件、增加召回数量，或者提示用户补充信息。如果还是不够，就给出保守回答。

问题：如果检索结果很多，系统该怎么做？ 答案：可以通过重排、过滤、摘要或只保留最相关片段来减少噪声。这样能避免上下文过长和无关信息干扰。

问题：什么是“召回”和“排序”？ 答案：召回是先尽量找出可能相关的内容，排序是再从这些内容中选出最相关的结果。一个好的检索系统通常要同时做好这两步。

问题：为什么在 RAG 系统里，检索质量往往比生成模型本身更重要？ 答案：因为生成模型主要负责组织语言，而答案的事实基础通常来自检索结果。如果检索内容不相关，模型即使表达得很好，最终答案也可能偏题或错误，所以检索质量常常是决定整体效果的关键。

问题：为什么 Chunk 切分时要考虑语义边界？ 答案：如果切分不考虑语义边界，片段可能在句子中间被截断，导致上下文丢失，影响 embedding 和检索效果。按标题、段落或自然语义分块通常更合理。

问题：向量检索和关键词检索分别适合什么场景？ 答案：向量检索更适合语义相近但词面不同的问题，关键词检索更适合技术名词、专有名词和精确匹配场景。技术面试知识库通常两者都需要。

问题：为什么很多检索系统会加 Rerank？ 答案：因为初步召回阶段主要追求覆盖率，可能会带来一些不够相关的候选。Rerank 可以在候选中进一步筛选，提升最终返回内容的相关性和准确性。

问题：什么情况下需要做 Query Rewrite？ 答案：当用户问题很口语化、很长、含有很多上下文或者表达不清楚时，Query Rewrite 往往有帮助。它能把问题改写成更适合检索系统处理的形式。

问题：什么是“检索到的内容要和问题强相关”？ 答案：这意味着返回的资料不只是表面关键词匹配，而是在语义上真正能够回答用户问题。强相关的内容通常能直接支持生成准确答案。

问题：为什么要在知识库中保留来源和页码信息？ 答案：来源和页码信息可以帮助追踪答案依据，也便于调试检索质量。如果答案有问题，可以快速定位到具体文档片段。

问题：什么是“上下文污染”？ 答案：上下文污染是指无关、重复或错误的信息混入模型输入，干扰回答质量。RAG 系统如果上下文组织不好，很容易出现这种问题。

问题：如何减少上下文污染？ 答案：可以通过更好的检索、更严格的筛选、重排、摘要压缩和上下文裁剪来减少无关内容。核心原则是只把最有用的信息交给模型。

问题：为什么结构化输出对工程实现很重要？ 答案：因为结构化输出更容易被程序解析、存储和展示，也更方便做后续的评分、统计和前端渲染。相比自由文本，它的稳定性更高。

问题：如果模型输出格式经常不稳定，怎么处理？ 答案：可以通过更严格的 prompt、给出示例、增加字段约束和后处理校验来改善。必要时还可以做重试机制。

问题：什么是“工具白名单”？ 答案：工具白名单是指 Agent 只能调用允许的工具，而不能随意访问外部能力。它能提高系统安全性和可控性。

问题：为什么 Agent 需要限制可调用工具？ 答案：因为如果不限制，模型可能调用不该调用的接口，造成安全和业务风险。限制工具范围是控制 Agent 行为的重要手段。

问题：什么是“任务状态”？ 答案：任务状态是指 Agent 当前处理到了哪一步、已经获得了什么信息、还需要什么信息。它是多步任务顺利执行的基础。

问题：为什么状态管理对多步 Agent 很重要？ 答案：因为多步任务需要持续记录中间结果，否则模型会忘记前面做过什么，导致重复操作或者流程中断。状态管理能让任务推进更稳定。

问题：什么是“失败重试”机制？ 答案：失败重试是指当工具调用、检索或模型输出失败时，系统重新尝试一次或多次。它能提升整体成功率，减少偶发错误带来的影响。

问题：为什么一些系统会设计“失败后改写再试”的流程？ 答案：因为第一次检索可能因为问法不合适而效果不好。改写查询后再检索，往往可以提高召回相关内容的概率。

问题：什么是“保守生成”？ 答案：保守生成是指在证据不足时尽量减少推断，优先输出已确认的信息。这样可以降低幻觉和错误回答的风险。

问题：为什么在信息不足时，保守生成比强行补全更好？ 答案：因为强行补全很容易生成错误内容，而保守生成至少不会把不确定信息说成事实。对于面试助手来说，准确性往往比完整性更重要。

问题：什么是“检索增强生成”的核心目标？ 答案：核心目标是让模型的回答建立在外部真实知识上，而不是只依赖模型参数记忆。这样可以提高准确性、时效性和可解释性。

问题：为什么知识库内容要持续清理和维护？ 答案：因为重复、过期或噪声内容会降低检索效果，影响最终回答质量。定期清理知识库可以保持召回准确性。

问题：什么是“增量入库”？ 答案：增量入库是指在已有知识库基础上，只添加新增或更新的内容，而不是每次全部重建。它更适合持续迭代的系统。

问题：为什么知识库版本管理有价值？ 答案：因为当知识库更新后，旧版本和新版本可能会影响检索结果。版本管理可以帮助定位问题，也方便回滚和对比效果。

问题：什么是“候选片段”？ 答案：候选片段是检索阶段先找出来的可能相关文本块。它们通常还需要经过重排、过滤或摘要，才能真正用于生成。

问题：为什么先召回候选片段再生成答案，而不是直接生成？ 答案：因为先召回候选片段能给模型提供可靠依据，减少凭空生成的风险。直接生成虽然快，但更容易不准确。

问题：什么是“答案依据不足”的情况？ 答案：答案依据不足是指检索到的资料和问题关系不强，或者资料本身不完整。这种情况下，系统应该谨慎输出，不宜过度推断。

问题：什么是“工程化的 Agent 应用”？ 答案：工程化的 Agent 应用不是只看模型能不能回答，而是看整个系统是否稳定、可维护、可扩展、可调试。它更重视真实可落地能力。

问题：为什么 Agent 项目中日志很重要？ 答案：因为日志可以记录工具调用、检索结果、模型输入输出和错误信息，方便排查问题。没有日志，Agent 很难调试和优化。

问题：什么是“可调试性”？ 答案：可调试性是指系统出问题后，能不能快速定位问题在哪一步出现。对于 Agent 来说，这通常依赖流程拆解、日志和状态记录。

问题：为什么要把复杂逻辑拆成多个模块？ 答案：因为这样更容易维护、测试和替换。比如检索、生成、评分和追问拆开后，哪一部分出问题都能更快定位。

问题：什么是“模块职责单一”？ 答案：模块职责单一是指每个模块只做一件事，比如只负责检索、只负责评分或只负责解析。这样更易复用，也更不容易出错。

问题：为什么过度复杂的 Agent 设计不一定更好？ 答案：因为复杂设计会增加延迟、成本和调试难度。对于实际业务来说，稳定、清晰、可维护往往比“看起来很高级”更重要。

问题：什么是“从知识库中找事实，再由模型组织语言”？ 答案：这就是很多 RAG 系统的基本思路。知识库负责提供事实基础，模型负责把这些事实整理成自然语言回答。

问题：为什么模型的语言能力不能替代知识检索？ 答案：因为模型语言能力强，不代表它掌握最新、最准确的外部知识。对于面试助手这种需要专业内容的场景，检索仍然是必需的。

问题：什么是“提示词约束输出范围”？ 答案：就是在 prompt 中明确告诉模型只回答哪些内容、不回答哪些内容。这样可以减少跑题、胡编和格式不一致的问题。

问题：为什么一个面试知识库最好同时有概念、原理和实现？ 答案：因为真实面试常常不只问定义，还会问底层机制和工程实现。资料覆盖得越全面，系统回答就越贴近真实面试需求。

问题：什么是“知识点可检索性”？ 答案：知识点可检索性是指一条知识是否容易被检索系统命中。短小、主题明确、结构清晰的知识点通常更容易被命中。

问题：为什么知识库里的内容不宜过长？ 答案：内容过长会让检索片段太重、噪声太多，也不利于精准召回。对向量检索来说，简洁且主题明确的文本往往更有效。

问题：什么是“语义独立”的知识点？ 答案：语义独立是指一条知识尽量只讲一个主题，不和太多其他概念混在一起。这样更利于切分、检索和回答复用。

问题：前端实现大模型流式输出时，SSE 和 WebSocket 的选型逻辑是什么？ 答案：如果主要需求是服务端单向持续推送文本，SSE 通常更简单，协议开销小，浏览器原生支持也较好；如果需要双向通信、实时交互、复杂事件交互，WebSocket 更合适。高并发场景下，SSE 结构更轻、实现成本更低，但 WebSocket 在连接管理和双向控制上更灵活。

问题：流式返回过程中网络中断、前端重连后，后端如何恢复上下文继续输出？ 答案：通常需要给每次流式请求绑定唯一的会话 ID 和生成任务 ID，并在后端保存已输出的偏移位置或状态快照。前端重连后，带上任务 ID 和断点信息，后端从上次未完成的位置继续推送，避免重复和丢失。

问题：用户点击“停止生成”后，后端如何立即终止 LLM 推理？ 答案：后端应为每个生成任务维护可取消控制信号或中断标记，在收到停止请求时立即触发取消。模型推理线程或协程需要定期检查中断状态，并尽快释放占用的 GPU、CPU 和上下文资源，避免继续无效计算。

问题：流式返回时，如何插入非文本事件而不影响前端渲染？ 答案：可以在流中定义统一事件协议，把文本片段、工具调用、错误信息、状态标记等都包装成结构化事件，例如 type + payload。前端根据事件类型分别处理，文本照常拼接，非文本事件走状态更新，不直接渲染为正文。

问题：多轮对话加流式输出时，如何保证消息不乱序？ 答案：每条消息和每段流式片段都要带上会话 ID、轮次 ID 和序列号。前端按轮次和顺序号拼接，后端按任务状态严格控制输出顺序，这样即使并发请求或网络抖动，也能保持上下文一致。

问题：高并发下大量 SSE 长连接如何做心跳检测和超时释放？ 答案：需要为连接设置心跳包和空闲超时机制，定期检测客户端是否仍然在线。服务端应限制单用户并发连接数，及时释放无效连接，并避免为每个连接长期占用过多线程或阻塞资源。

问题：流式输出场景中，如何实现内容安全实时截断？ 答案：可以在流式生成的每个片段上做实时敏感内容检测，一旦命中规则立即终止输出。与此同时，要记录触发原因并清理上下文状态，防止后续继续生成不合规内容。

问题：流式返回时，如何精准统计 Token 消耗？ 答案：可以在每段输出时记录增量 token 数，再在结束时汇总总消耗。若模型接口本身支持 usage 回传，最好结合每次调用结果统一统计，便于计费和监控。

问题：小程序、APP、PC 端对流式输出的兼容性差异如何处理？ 答案：不同端对 SSE、WebSocket 和长连接的支持程度不一样，通常需要在客户端层做适配封装。对于渲染卡顿问题，可以控制前端增量更新频率，避免每个 token 都强制刷新 UI。

问题：流式推理时模型中途报错，如何设计兜底策略？ 答案：需要在后端捕获异常并返回明确的错误状态，同时保留已生成内容。前端可以提示“生成中断，是否重试”，后端则根据任务状态决定是否从断点恢复或重新生成。

问题：如何实现流式输出的断点续打？ 答案：关键是保存已生成内容、当前位置和会话状态。用户刷新页面后，前端携带会话标识重新建立连接，后端根据保存的生成进度继续推送，而不是从头生成。

问题：跨服务流式透传时，如何做日志埋点？ 答案：应在每段输出、每次工具调用、每次异常和每个关键状态点打埋点，并带上统一 trace_id。这样可以串联 Java/Go 后端和 Python 模型服务的全链路耗时与异常信息。

问题：Agent 执行环路 Plan→Act→Observe→Reflect 在生产中如何落地？ 答案：可以把规划、执行、观察和反思拆成明确步骤，并为每一步定义输入输出和异常处理。规划负责拆解任务，执行负责调用工具，观察负责读取结果，反思负责判断是否需要重试或调整策略。

问题：ReAct 框架在实际开发中如何避免思考与行动脱节？ 答案：要让 Reason 阶段尽量基于当前观测到的信息，而不是脱离工具结果空想。实践中通常会限制每轮思考必须引用当前上下文和工具返回结果，再决定下一步动作。

问题：Agent 工具调用的 Schema 设计核心是什么？ 答案：核心是让工具的名字、用途、参数类型和返回结构都足够清晰，便于模型稳定选择和调用。参数越明确，模型越不容易出现缺参、类型错或语义歧义。

问题：多步工具依赖如何设计，避免重复调用和死循环？ 答案：要给每一步设置明确的状态标记和完成条件，已经完成的步骤不再重复执行。对于依赖链较长的任务，可以记录中间结果，只有在输入变化或结果失效时才重新触发相关步骤。

问题：Agent 的反思机制如何实现？ 答案：可以在每轮执行后，让模型对结果进行检查，判断是否满足目标、是否存在错误或遗漏。若发现问题，再根据失败原因调整下一轮计划或重新调用工具。

问题：Agent 的短期记忆和长期记忆如何设计？ 答案：短期记忆通常保存当前会话的最近上下文和任务状态，长期记忆保存跨会话的用户偏好、历史结论或重要事实。实现上可用不同存储层级，结合摘要和检索控制容量。

问题：高并发场景下，Agent 任务排队、限流、优先级调度如何实现？ 答案：可以按用户等级、任务类型或业务优先级进入不同队列，再由调度器统一控制并发。付费用户、关键业务或高优先级任务可优先处理，普通任务则排队等待。

问题：Agent 调用工具超时后如何设计重试、熔断和降级？ 答案：短时失败可进行有限次数重试，连续失败则触发熔断，暂停对该工具的调用。降级时应返回保守且友好的提示，比如说明当前服务繁忙，并给出替代操作或稍后重试建议。

问题：Agent 生成的 SQL 或代码需要执行时，如何保证安全？ 答案：必须在沙箱环境中执行，并限制文件系统、网络和数据库权限。执行前还要做输入校验和权限隔离，避免越权操作、注入攻击和破坏性命令执行。

问题：LangChain 和 LangGraph 在生产中如何选型？ 答案：如果业务流程相对简单，LangChain 足够；如果涉及复杂状态流转、分支、重试和人工干预，LangGraph 更适合。LangGraph 更像流程控制框架，适合复杂业务编排。

问题：Agent 执行过程如何做可观测？ 答案：需要记录每一步的输入、输出、工具调用、耗时、token 消耗和异常信息，并统一关联 trace_id。这样可以在生产中快速定位问题和评估性能。

问题：多个用户同时触发同一个 Agent 任务时，如何做幂等设计？ 答案：可以通过任务 ID、请求 ID 或业务唯一键做去重，确保同一任务不会被重复执行。对于外部接口和数据库操作，也要设计幂等写入和幂等查询机制。

问题：Agent 如何安全传递用户身份并调用工具？ 答案：身份信息应通过受控上下文或服务间认证令牌传递，不要直接暴露给模型生成内容。工具调用时要做权限校验，确保模型只能在授权范围内操作。

问题：如何实现 Agent 执行过程的可回放、可打断、可人工干预？ 答案：要把每一步执行状态持久化，并支持从中间状态恢复。人工干预时，系统可以暂停当前任务，由人工接管关键步骤，确认后再继续执行。

问题：Agent 的任务分解能力如何优化？ 答案：可以通过更清晰的提示词、固定的分解模板和任务示例来引导模型拆解步骤。对于复杂任务，还可以在规划阶段先生成子任务列表，再逐个执行和校验。

问题：开源 Agent 框架在生产中落地有哪些常见问题？ 答案：常见问题包括流程不稳定、工具调用不规范、状态管理复杂、日志不完整和性能开销偏大。落地时通常需要在框架之上增加更多工程约束和监控能力。

问题：Agent 与现有后端系统对接时，如何保证接口调用稳定性？ 答案：要明确接口契约、超时重试策略、错误码规范和幂等要求。与此同时要做好熔断、降级和监控，避免单点故障影响整个 Agent 流程。

问题：如何评估 Agent 的任务完成率？ 答案：可以统计任务是否成功达成目标、步骤是否完整、工具调用是否合理、异常是否可恢复等指标。生产上常结合成功率、失败率、平均步骤数和人工抽检结果综合评估。

问题：百万级文档 RAG，如何设计索引、分片、缓存架构？ 答案：通常会先按业务域和文档类型分片，再在每个分片内建立向量索引和必要的关键词索引。线上查询时优先走缓存和分片路由，只把请求发到最相关的分片，减少检索范围和延迟。

问题：文档频繁更新或删除时，向量库如何保证实时一致性？ 答案：一般需要给文档打版本号和状态标记，删除或更新时先标记失效，再异步清理索引。查询阶段要过滤掉已失效版本，避免召回旧知识或脏数据。

问题：同一用户同一问题多次查询时，如何做检索结果缓存？ 答案：可以对查询文本、用户身份和知识库版本做缓存键，命中后直接返回已排序的候选结果。为了避免缓存过期与新知识更新冲突，通常需要绑定知识库版本号或设置较短 TTL。

问题：用户问题模糊时，如何提升召回率？ 答案：可以先做意图识别，再进行查询改写，并同时走关键词、语义和历史上下文多路召回。最后再把候选结果合并重排，这样通常比单一路径更稳。

问题：表格、带格式 PDF、图片文本的 RAG 应该怎么处理？ 答案：重点是尽量保留结构信息，比如表格行列关系、标题层级和图片中的 OCR 文本。对于复杂 PDF，常需要结合版面解析、OCR 和结构化抽取，而不是直接按纯文本读取。

问题：长文档 RAG 出现中间内容丢失，怎么解决？ 答案：可以用父子分块、分层检索和重排序来缓解。先用小块做召回，再回到父块或更大的上下文里补充信息，这样更容易覆盖长文档中的关键内容。

问题：混合检索在生产中如何调参？ 答案：通常要根据业务数据观察关键词召回和向量召回的比例，再调节两者权重、召回数量和重排阈值。目标是在召回率和延迟之间找到平衡，不是单纯追求某一项最优。

问题：RAG 与多轮对话结合时，如何实现基于历史上下文的自动检索？ 答案：可以先把当前轮问题和历史摘要结合成检索查询，再做一次相关性判断，避免对每一轮都重复检索。必要时还可以把上一轮检索结果缓存一段时间，减少无效调用。

问题：如何避免 RAG 检索到大量无关片段？ 答案：常见做法包括更好的切分、召回后重排、关键词过滤、查询改写和上下文压缩。核心是减少噪声，把最相关的少量片段交给模型。

问题：生产环境如何自动化评估 RAG 效果？ 答案：通常会准备一批标准问答集，离线评估召回率、MRR、答案相关性等指标，再结合人工抽检看最终回答质量。线上还可以记录点击、追问和用户反馈，形成闭环。

问题：Embedding 模型如何选型？ 答案：要结合语言类型、领域语料和性能需求来选。中英文混合场景通常要优先选择对多语言支持较好的模型，同时平衡召回效果和推理速度。

问题：向量检索中余弦相似度、点积、欧氏距离有什么差异？ 答案：余弦相似度更关注方向，常用于文本语义相似；点积常和向量归一化配合使用；欧氏距离更常见于几何意义更强的场景。文本检索里通常优先考虑余弦相似度或归一化后的点积。

问题：RAG 的 rerank 模型怎么选？ 答案：一般会优先选和业务语言风格接近、对相关性判断能力强的模型。实际落地时要同时看重排效果和推理速度，因为 rerank 常常是线上延迟的重要来源。

问题：动态知识更新场景下，RAG 如何做增量更新？ 答案：可以对新增、修改和删除文档分别处理，只对变更部分重新切分和向量化。这样比全量重建更高效，也更适合每日持续更新的知识库。

问题：如何处理 RAG 中的噪声文档？ 答案：常见做法包括去重、过滤无效文档、清理低质量内容、设置文档白名单和黑名单。噪声太多会明显拉低检索和生成质量，所以入库前清洗很重要。

问题：RAG 服务高并发下如何做负载均衡和水平扩展？ 答案：可以把检索、重排和生成拆成独立服务，通过负载均衡器分发请求。向量检索层和生成层都可以水平扩展，缓存和限流也能帮助平稳峰值流量。

问题：RAG 与 Fine-tuning 在生产中如何选型？ 答案：如果知识经常变化、需要引用外部资料，通常优先 RAG；如果任务模式稳定、输出风格固定，可以考虑微调。很多场景下两者会组合使用，而不是二选一。

问题：如何解决 RAG 的检索滞后问题？ 答案：要缩短入库到可检索之间的链路，尽量做到增量更新、快速索引刷新和版本切换。查询时也要避免缓存过久，把新知识及时暴露给检索层。

问题：峰值 QPS 很高时，LLM 接口如何做排队和削峰？ 答案：可以把请求放进队列，按优先级和资源情况批量调度，并设置超时和最大并发限制。对于大模型推理服务，批量推理和动态批处理通常能明显提升吞吐。

问题：vLLM、TGI、TensorRT-LLM 的落地差异是什么？ 答案：它们都面向推理加速，但侧重点不同：有的更强调吞吐和连续批处理，有的更偏部署和工程集成，有的更偏极致性能优化。选型通常取决于模型规模、部署环境和对延迟或吞吐的要求。

问题：模型量化的原理是什么？ 答案：量化是用更低精度的数据类型表示模型权重或激活值，以减少显存占用和提高推理速度。常见有 INT8、INT4 等，但精度和稳定性可能会受到影响。

问题：模型 API 突然报错、限流或宕机时怎么处理？ 答案：通常要设计熔断、重试和备用模型切换机制。主模型异常时快速切到备用方案，同时对用户返回尽量平滑的降级提示，避免体验中断。

问题：如何防止恶意用户刷 Token 造成成本暴增？ 答案：可以在网关层做限流、配额控制、身份校验和异常行为检测。对超长上下文和高频请求也要设置约束，必要时直接拒绝或降级处理。

问题：生产中如何实现多模型调度？ 答案：可以根据任务复杂度和成本，选择小模型处理简单任务，大模型处理复杂推理。调度策略通常结合意图分类、置信度和业务优先级来决定调用哪个模型。

问题：GPU 资源有限时，如何做资源隔离和优先级控制？ 答案：可以通过队列隔离、配额限制和任务优先级调度来实现。高优先级任务优先获得 GPU 资源，低优先级任务则排队或降级到小模型。

问题：异步 Agent 任务执行时间很长时如何设计？ 答案：通常要把任务提交、执行、查询结果分离，任务状态持久化到数据库或队列系统中。执行完成后可以通过轮询、回调或消息通知把结果返回给用户。

问题：如何实现 LLM 请求的全链路压测？ 答案：需要模拟真实对话流量、流式请求和并发场景，并统计延迟、吞吐、错误率和资源使用情况。压测时要尽量贴近真实业务，而不是只压单一接口。

问题：微服务架构下，AI 模块与 Java/Go 后端如何保证最终一致性？ 答案：通常通过消息队列、事件驱动和状态机来处理跨服务流程。对于非强一致业务，最终一致性更适合 AI 场景，也更容易扩展。

问题：Agent 服务如何高可用部署？ 答案：一般要做多实例部署、负载均衡、健康检查和故障切换。关键组件还要支持降级和灾备，避免单点故障影响整条链路。

问题：LLM 结果、Embedding、检索结果的缓存如何设计？ 答案：可以按不同层级设置缓存，比如模型结果缓存、向量检索缓存和重排缓存。缓存时间要结合知识更新频率和业务时效性来定，更新快的内容 TTL 应该更短。

问题：日志量很大时，如何做存储和审计？ 答案：可以把热日志、冷日志和审计日志分层存储，热数据用于排障，冷数据用于归档。常见做法是接入日志系统做检索和分析，同时设置保留周期和压缩策略。

问题：如何做权限控制，避免 Agent 越权操作？ 答案：要把数据权限、工具权限和功能权限分开控制，并在服务端做严格校验。模型只能在授权范围内调用工具，不能直接接触敏感资源。

问题：Docker 和 K8s 部署 Agent/LLM 服务需要注意什么？ 答案：需要合理配置 CPU、内存和 GPU 资源请求，做好健康检查、滚动更新和自动扩缩容。对于大模型服务，还要特别关注显存和启动时间。

问题：MLOps 如何应用在 Agent 系统中？ 答案：可以用于模型版本管理、实验跟踪、数据版本控制和 CI/CD 流水线。这样模型更新、提示词调整和检索策略变更都能更有秩序地迭代。

问题：Prompt Injection 的生产级防御方案有哪些？ 答案：常见手段包括输入过滤、系统提示词隔离、工具白名单、输出审查和安全规则约束。核心原则是不要让用户输入直接覆盖系统指令。

问题：如何防止敏感信息进入 LLM？ 答案：可以在请求进入模型前先做脱敏或掩码处理，比如隐藏手机号、身份证号和订单号。必要时只把必要字段传给模型，减少泄露风险。

问题：如何控制 Token 消耗和模型成本？ 答案：可以通过限流、缓存、缩短上下文、选择合适模型和批量推理来控制。很多时候，成本优化比单纯追求大模型效果更重要。

问题：私有部署和 API 调用如何选型？ 答案：API 调用部署快、运维轻，但在成本、隐私和稳定性上受外部服务影响较大；私有部署更可控、更安全，但成本和运维复杂度更高。通常要结合业务敏感度和预算来选。

问题：如何保证 Agent 行为可审计和可回溯？ 答案：需要记录输入、输出、工具调用、状态变化和关键决策点，并保留 trace_id 关联全链路。这样出问题时可以回放整个执行过程。

问题：设计一个企业内部知识库问答 Agent，核心流程是什么？ 答案：一般是先做问题理解和权限判断，再检索知识库，必要时做重排和摘要，最后生成回答并保留来源。流程要尽量短、稳定、可追踪。

问题：设计一个 Text2SQL Agent 时，最关键的风险是什么？ 答案：主要风险是 SQL 注入、表结构理解错误和复杂查询生成不准确。通常要结合 schema 提示、权限控制、SQL 校验和执行前审核来降低风险。

问题：设计客服多 Agent 系统时，如何衔接意图识别、检索和人工转接？ 答案：可以先由意图识别 Agent 判断问题类型，再把请求分发给知识库检索、工单生成或人工转接模块。各 Agent 之间通过统一状态和事件流衔接，保证流程连续。

问题：如何设计一个低延迟、高并发的 RAG 服务？ 答案：核心是优化检索索引、减少无效召回、使用缓存、控制上下文长度并把重排和生成分层处理。百万级文档场景下，分片和负载均衡也很重要。

问题：多 Agent 协作系统如何解决冲突和通信问题？ 答案：需要明确各 Agent 的职责边界和通信协议，避免多个 Agent 同时修改同一状态。一般会有一个协调者负责仲裁和合并结果。

问题：从后端工程师角度，搭建可上线的 Agent 平台要包含哪些模块？ 答案：通常包括模型层、工具层、检索层、状态管理、任务队列、日志监控、权限控制和缓存层。平台化后还要支持版本管理和可观测性。

问题：如何做输出内容审核，避免 Agent 生成违规内容？ 答案：可以在输出前做安全过滤和规则审核，必要时结合内容分类模型进行二次判断。对于高风险场景，还应增加人工复核或强制降级策略。

问题：灾备设计中，主模型或主服务挂了怎么办？ 答案：要准备备用模型和备用服务，并设计自动切换流程。用户请求应尽量无感知地迁移到备用链路，避免长时间不可用。

问题：如何设计自动化运维 Agent？ 答案：通常会让它具备日志读取、问题定位、命令执行和告警通知能力，但必须放在权限受控的沙箱中。执行高风险操作前最好有人工确认。

问题：多模型服务网格如何设计？ 答案：可以通过统一网关管理不同模型实例，提供健康检查、路由切换和负载均衡能力。这样在模型升级、降级或故障时能更平滑地切换。

问题：如何处理 Agent 执行中的长尾任务？ 答案：长尾任务通常要单独隔离，设置更严格的超时和资源上限。必要时可以异步处理、分段执行或交给专门的低优先级队列，以控制整体成本。

问题：你了解过市面上有哪些智能体 Agent 吗？ 答案：市面上常见的 Agent 或 Agent 框架包括 AutoGPT、OpenAI 的工具调用型 Agent、LangChain Agent、LangGraph、Microsoft AutoGen、CrewAI、MetaGPT 等。它们大多围绕“任务分解、工具调用、状态管理、多轮执行”来设计，只是侧重点不同，有的偏通用编排，有的偏多智能体协作，有的偏工程化工作流。

问题：Agent 项目开发常用的框架有哪些？ 答案：常用框架主要有 LangChain、LangGraph、AutoGen、CrewAI、Semantic Kernel 等。实际选型时通常会看任务是否需要复杂状态流转、是否需要多 Agent 协作、是否需要和企业后端系统深度集成。

问题：你了解过哪些大模型？ 答案：常见大模型包括 GPT 系列、Claude、Gemini、Llama、Qwen、GLM、DeepSeek、Mistral 等。它们在上下文长度、推理能力、中文能力、工具调用支持和部署成本上各有差异。

问题：RAG 系统一般流程是什么？ 答案：RAG 的标准流程通常包括文档加载、清洗、切分、向量化、索引构建、查询检索、重排、上下文拼接和最终生成。核心目标是让模型回答建立在检索到的真实资料基础上。

问题：MCP 和 Function Calling 有什么区别？ 答案：Function Calling 更像是模型调用单个函数的机制，重点在于结构化参数和函数执行。MCP 更像一套标准化的上下文与工具接入协议，强调模型、工具和外部资源之间的统一通信方式。

问题：如何写好的 Prompt？ 答案：好的 Prompt 通常要包含清晰任务目标、角色设定、输入边界、输出格式和必要示例。对于结构化任务，最好明确要求输出字段，并尽量减少歧义和口语化描述。

问题：多轮对话一般怎么实现？ 答案：多轮对话通常通过会话 ID、消息历史、摘要记忆和上下文拼接来实现。系统会保留最近的对话内容，并在必要时对历史进行压缩或检索，以控制上下文长度。

问题：Agent 项目背景一般怎么介绍？ 答案：可以从“为什么需要 Agent”来讲，比如传统大模型只能回答问题，但在复杂任务里需要检索、规划、执行和反馈，所以要引入 Agent。项目背景通常要强调自动化、工具调用、知识增强和业务落地价值。

问题：LLM 产生幻觉的原因是什么？ 答案：幻觉通常来自信息不足、上下文不完整、训练分布限制或模型过度补全。模型会倾向于生成看起来合理的内容，但不一定有事实依据。

问题：LLM 幻觉有哪些常见解决方案？ 答案：常见方案包括 RAG 检索增强、限制回答边界、提示词约束、结果校验、来源引用和在不确定时保守回答。实际工程里往往要组合使用，而不是只靠一种方法。

问题：MCP 协议的核心内容是什么？ 答案：MCP 的核心是标准化模型与外部能力的接入方式，包括资源、工具、提示词等内容的统一管理和调用。它的目标是降低模型接入工具、知识源和上下文的集成成本。

问题：推理模式的差异化设计是什么？ 答案：推理模式差异化设计是指根据任务复杂度、成本和时延要求，选择不同的推理路径。比如简单任务可以直接回答，复杂任务则走检索、规划和多步执行流程。

问题：RAG 检索怎么优化？ 答案：常见优化手段包括更合理的切分、混合检索、查询改写、重排、分层召回和上下文压缩。目标是提高相关性，同时控制延迟和噪声。

问题：特定推理模型不支持 MCP 的技术原因是什么？ 答案：通常是因为模型或服务端没有实现 MCP 所需的协议适配、工具注册或上下文管理能力，也可能是产品层没有开放对应接口。换句话说，这更多是协议接入和工程实现问题，而不只是模型能力问题。

问题：Agent 推理模式有哪些？ 答案：常见推理模式包括 ReAct、Plan-and-Execute、Chain-of-Thought、Tree-of-Thought、Reflection 等。它们在“先思考后执行”还是“边思考边执行”、是否有反思回路上有所不同。

问题：如何构建跨模块错误追踪的 Agent 知识库？ 答案：可以把错误日志、工具调用失败案例、检索失败样本、模型异常输出和修复方案统一沉淀到知识库中，并附带模块标签、错误类型和处理结果。这样 Agent 或排障系统就能基于历史案例快速定位问题。

问题：多 Agent 执行策略如何智能选择和切换？ 答案：通常需要一个协调器根据任务类型、当前状态和 Agent 能力进行路由。比如简单问题交给单 Agent，复杂流程拆给多个专业 Agent，并在执行中根据结果动态切换。

问题：简历关键词提取怎么做？ 答案：一般会结合规则抽取和 NLP 方法，先从技能、项目、岗位名称、技术栈等字段中识别关键词，再做去重和归一化。对于更复杂场景，还可以用大模型做语义抽取和补全。

问题：RAG 评估方案怎么设计？ 答案：通常要同时评估检索和生成两部分，检索侧看召回率、MRR、NDCG，生成侧看答案相关性、正确性和可读性。最好再结合人工抽检和真实用户反馈形成闭环。

问题：SSE 的局限性是什么？ 答案：SSE 主要适合单向服务端推送，双向交互能力弱，且在某些环境下对连接保活、代理转发和移动端兼容会有局限。对于需要双向实时通信的场景，WebSocket 往往更灵活。

问题：复杂任务下执行流程一般怎么举例？ 答案：比如“根据简历和 JD 生成面试题并评分”这类任务，通常会先解析简历和 JD，再提取关键信息，然后检索知识库，接着生成题目、评分答案、再生成追问。每一步都可以单独记录和校验。

问题：MCP 通信方式有哪些？ 答案：MCP 通常通过标准化消息和协议层进行通信，常见会结合本地进程、网络服务或标准输入输出等方式接入。核心重点是统一工具、资源和上下文的交互方式。

问题：项目中 AI 贡献的代码占比怎么理解？ 答案：通常是指在整个项目里，由 AI 辅助生成、改写或重构的代码比例。更重要的是说明 AI 在哪些环节帮助了你，比如脚手架、接口封装、测试、文档和提示词开发。

问题：Prompt 工程有什么实践经验？ 答案：实践上最重要的是明确任务目标、限制输出格式、提供少量高质量示例，并不断根据错误样本迭代。对于抽取和分类类任务，结构化约束通常比长篇描述更有效。

问题：基于代码构建知识库的 Agent 怎么设计？ 答案：可以先对代码仓库做解析，提取文件结构、函数、类、注释和调用关系，再切分成适合检索的知识单元。随后结合代码问答、缺陷排查和重构建议等能力，构建代码知识 Agent。

问题：A2A 协议是什么？ 答案：A2A 一般指 Agent-to-Agent 协议，强调不同 Agent 之间的标准化通信和协作方式。它的目标是让多个 Agent 能够在统一规则下交换任务、状态和结果。

问题：长文本生成有哪些技术方案？ 答案：常见方案包括分段生成、提纲先行、检索增强、上下文压缩和分层生成。对于特别长的内容，还可以先生成大纲，再逐节展开，减少跑题和前后不一致。

问题：什么是 Agent skills？ 答案：Agent skills 可以理解为 Agent 可复用的能力模块，比如检索、总结、代码生成、SQL 生成、文件读取等。把能力封装成 skill 后，Agent 更容易按任务组合调用。

问题：如何演示 Agent 项目实现细节？ 答案：演示时通常要展示任务输入、Agent 判断、工具调用、检索结果、最终输出和日志链路。最好能通过一条完整任务链路说明系统是如何一步步工作的。

问题：你了解其他 Agent 范式吗？ 答案：除了 ReAct 和 Plan-and-Execute，还有 Reflection、Tree-of-Thought、Multi-Agent Collaboration、Workflow Agent 等范式。它们主要区别在于是否多轮反思、是否多 Agent 协作、是否强调固定流程编排。

问题：模型预热机制是什么？ 答案：模型预热是指在正式接收流量前先加载权重、初始化缓  、建立推理图或执行一次空请求。这样可以减少首个请求的冷启动延迟。

问题：NL2SQL 场景下如何做 SQL 安全防护？ 答案：一般要限制可访问表和字段、对生成 SQL 做语法校验和权限校验，并禁止危险语句。必要时还要加执行前审核和只读沙箱，避免注入和误操作。

问题：如何提升复杂任务执行准确率？ 答案：可以通过任务拆解、工具约束、反思机制、结果校验和错误案例迭代来提升。对于关键步骤，还可以增加人工确认或二次验证。

问题：AI 辅助 IDE 开发工具有哪些？ 答案：常见的有 GitHub Copilot、Cursor、Codeium、Continue 等。它们主要帮助代码补全、重构、解释代码和生成测试。

问题：RAG 动态知识更新怎么做？ 答案：一般采用增量入库、版本控制和索引刷新机制，只更新变更文档，不做全量重建。这样可以兼顾实时性和性能。

问题：MCP 和 skill 有什么区别？ 答案：MCP 更偏协议层，负责标准化接入工具和资源；skill 更偏能力层，是 Agent 可调用的具体功能模块。前者管“怎么连”，后者管“能做什么”。

问题：推理模式怎么选择？ 答案：要根据任务复杂度、成本、时延和准确率要求来选。简单问答可以直接生成，复杂任务适合走检索、规划、多步执行和反思流程。

问题：企业内部知识库 RAG 的动态持续更新方案怎么设计？ 答案：可以结合文档变更监听、增量切分、异步向量化和索引刷新实现持续更新。查询时再结合版本号和缓存控制，避免读到旧内容。

问题：Prompt 设计可以举个示例吗？ 答案：比如让模型做结构化抽取时，可以明确要求“只输出 JSON，不要解释”，并给出字段名、字段含义和示例格式。这样输出会更稳定，也更容易被程序处理。

问题：A2A 与 MCP 有什么区别？ 答案：A2A 关注的是 Agent 与 Agent 之间的协作通信，MCP 更关注模型如何标准化接入外部工具和上下文资源。一个偏协作协议，一个偏工具接入协议。

问题：多阶段召回策略怎么优化？ 答案：通常会先用粗召回扩大覆盖，再用精召回和重排筛选候选，必要时还会加过滤和上下文压缩。这样可以在召回率和速度之间取得平衡。

问题：AI 辅助开发有哪些实践经验？ 答案：比较有效的做法是把 AI 用在脚手架、重复代码、文档、测试和重构上，而不是完全依赖它写关键业务逻辑。对复杂逻辑要结合人工审查和单元测试，确保质量。
